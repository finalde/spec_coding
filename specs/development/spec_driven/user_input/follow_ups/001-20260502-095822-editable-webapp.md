# Follow-up draft 001 — 2026-05-02

Summary: Drop the strictly-readonly constraint. The viewer becomes an interactive editor for every artifact in its exposed tree, surfaces a colored-block Q&A view for `interview/qa.md`, and provides per-stage and per-project regeneration-prompt panels with an autonomous-mode toggle. The autonomous-mode contract is recorded in `CLAUDE.md`.

## New / changed requirements

### 1. End the readonly constraint

- Every file currently rendered by the viewer is now also **editable in place**.
- Editing UX target: friendly enough that a non-developer can patch a Q/A or spec section without touching a text editor outside the app.
- Default mode is "view"; an explicit ✎ Edit toggle on the file pane switches to an editor for the same file. Save / Discard / Close-editor controls are visible. Save calls a new `PUT /api/file` that uses the same sandbox/path rules as the existing reader.

### 2. Colored-block Q&A view for `interview/qa.md`

- When the active file is an `interview/qa.md`, render the structure (`## Round N` → `### category` → `**Q:** …` / `- A: …`) as discrete, color-differentiated blocks instead of a flat markdown render.
- Question blocks use one tint, answer blocks use another, the category gets a badge.
- Each Q and each A is independently editable in place — clicking pencil on a single Q or A opens a small editor scoped to that block, with Save persisting back to the same `qa.md` file via `PUT /api/file`.
- Plain markdown editing of the whole `qa.md` remains available via the file-level Edit toggle.

### 3. Per-stage "Regenerate prompt" panel

- On every stage file, surface a collapsible "Regenerate" panel that lets the user kick off a copy-paste regeneration prompt for that stage.
- Panel contents:
  - Module checkboxes for the artifacts produced by that stage (default: all checked).
  - "Autonomous mode" toggle (persisted in `localStorage`, default off).
  - An expandable widget that shows the assembled copy-paste prompt with a Copy button.
- Prompt assembly happens server-side at `POST /api/regen-prompt` and pulls in the current `revised_prompt.md` plus all `follow_ups/*.md` so the user is always copying the freshest intent.

### 4. Project parent page with master regen panel

- New route: `/project/{type}/{name}` showing the project's stage map and a single master regeneration panel.
- The master panel lets the user select **which stages and which modules to regenerate** (default: all). It produces a single combined copy-paste prompt that walks the chosen stages in order.
- Same autonomous-mode toggle as the per-stage panel; same `localStorage` setting drives both.

### 5. Autonomous-mode contract (documented in `CLAUDE.md`)

- The pasted regen prompt opens with one of two headers:
  - `# EXECUTION MODE: AUTONOMOUS` — Claude must NOT call `AskUserQuestion`, must use best judgment for any ambiguity (recording the choice inline in the artifact), and must produce every requested artifact in a single uninterrupted run before stopping. Use case: user kicks off a full regen and walks away.
  - `# EXECUTION MODE: INTERACTIVE` — default behavior, may use `AskUserQuestion`.
- This contract is documented under a new `## Regeneration prompts & autonomous mode` section of `CLAUDE.md` so future Claude sessions interpret pasted regen prompts correctly.

### 6. Backend additions

- `PUT /api/file` — sandboxed write to a path inside the existing exposed tree, atomic-replace via temp file + rename, same size/extension/symlink rules as `GET /api/file`.
- `GET /api/stages?project_type=&project_name=` — returns the canonical six-stage definition with per-stage module list (drives the regen panel UI).
- `POST /api/regen-prompt` — body `{project_type, project_name, stages[], modules{}, autonomous}` → returns `{prompt}` ready to paste into Claude Code.

## Out of scope for this follow-up

- No "run Claude inline" — regeneration prompts are still copy-paste into Claude Code CLI. The webapp does not call out to the model.
- No diff / version history beyond what git already provides.
- No live file watcher / auto-refresh.
- No multi-user, no auth — localhost-only stays per the existing security model.
