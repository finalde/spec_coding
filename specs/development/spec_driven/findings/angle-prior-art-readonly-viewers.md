# Angle — Prior art for read-only viewers of structured artifact trees

The `spec_driven` viewer/editor exposes a fixed tree of markdown + YAML + JSON artifacts (CLAUDE.md, agent files, skill files, and per-project five-stage subfolders under `specs/{task_type}/{task_name}/`). The reading experience it most resembles is a documentation site or a code-host file browser, not a wiki. This angle surveys eight mature web tools that solve the "render a tree of structured files for browsing" problem, focusing on sidebar grouping, expand/collapse defaults, breadcrumbs, and deep-link URL behavior. It then distills 5–8 concrete adopt/skip recommendations for the spec_driven sidebar + reader pattern.

## 1. GitHub web file browser

GitHub's repo file view uses the URL pattern `/{owner}/{repo}/blob/{branch}/{path}` for files and `/tree/{branch}/{path}` for directories. The file tree renders as a left sidebar adjacent to the blob viewer, and "improved file navigation and editing in the web UI" (Sept 2025) added a "View on default branch" button left of the branch picker that hides on the default branch, disables when the file does not exist on the default branch, and otherwise jumps directly to that file on default ([GitHub Changelog 2025-09-04](https://github.blog/changelog/2025-09-04-improved-file-navigation-and-editing-in-the-web-ui/)).

- **Sidebar grouping:** raw filesystem hierarchy. No semantic categories; folder names are the categories.
- **Expand/collapse:** ancestors of the currently viewed file auto-expand; siblings stay collapsed.
- **Breadcrumbs:** path segments above the blob viewer are clickable, each linking to the corresponding `/tree/...` URL.
- **Deep links:** `#L42` and `#L42-L60` line ranges; copyable from line gutters.
- **Anti-pattern to skip:** GitHub's tree does not group files by purpose — for our exposed tree, surfacing only the "five canonical stage subfolders" inside a project is more legible than dumping the raw `specs/{type}/{name}/` listing as siblings.

## 2. Sourcegraph code search

Sourcegraph's blob view exposes Hover popovers ("Hovers allow you to quickly glance at the type signature and accompanying documentation of a symbol definition without having to context switch"), Jump to definition, Find references in a bottom panel, plus utility actions including **Open in Editor**, **View History**, **Blame**, **Open in CodeHost**, **Permalink** (copy shareable link to specific line ranges), **Raw download**, and **Line Wrapping** ([Sourcegraph code navigation features](https://sourcegraph.com/docs/code-search/code-navigation/features)).

- **Sidebar grouping:** repo + file tree, like GitHub, but with cross-repo navigation overlays.
- **Deep links:** Permalink action explicitly produces a "shareable link to specific line ranges" — a copy-link affordance, not just URL inspection.
- **Anti-pattern to skip:** symbol-level navigation (Go to definition, Find references) is overkill for markdown/YAML/JSON; spec_driven artifacts are read by humans, not parsed for symbols.

## 3. MkDocs Material navigation

Material for MkDocs exposes a rich set of navigation flags ([MkDocs Material navigation setup](https://squidfunk.github.io/mkdocs-material/setup/setting-up-navigation/)):

- `navigation.instant` intercepts internal links via XHR for SPA-style transitions; `navigation.instant.prefetch` starts fetching on hover; `navigation.instant.progress` shows a top progress bar.
- `navigation.tracking` updates the address bar with the active anchor as the user scrolls.
- `navigation.tabs` renders top-level sections as a menu layer below the header; `navigation.tabs.sticky` keeps it visible during scroll.
- `navigation.sections` renders top-level sections as sidebar groups (header rows, not collapsible nodes).
- `navigation.expand` "expand[s] all collapsible subsections by default."
- `navigation.path` "Renders breadcrumb navigation above the page titles, which might make orientation easier for users visiting your documentation on devices with smaller screens."
- `navigation.indexes` lets a section node also be a content page.
- `navigation.prune` ships only visible nav items in HTML, "reducing site size by 33%+."
- `toc.follow` auto-scrolls the right-side TOC so the active anchor stays visible.

- **Sidebar grouping:** explicit `nav:` block in `mkdocs.yml`, manually authored.
- **Expand/collapse default:** sections are collapsed by default; `navigation.expand` flips the default to expanded.
- **Anti-pattern to skip:** Material's `tabs.sticky` adds a second navigation layer above the sidebar — for an opinionated 3-pane viewer, a single tree is clearer than tabs + tree + TOC.

## 4. Docusaurus docs sidebar

Docusaurus supports two sidebar generation modes: **autogenerated** (filesystem-driven via `_category_.json`/`_category_.yml` files) and **manual** (explicit JS array). The `collapsible: true` and `collapsed: false` properties on a category control collapsibility and default open state. The global option `themeConfig.docs.sidebar.autoCollapseCategories` "would collapse all sibling categories when expanding one category. This saves the user from having too many categories open and helps them focus on the selected section." `themeConfig.docs.sidebar.hideable` lets the entire sidebar be hidden, "allowing users to better focus on the content," especially on tablet-width screens. Breadcrumbs render the sidebar path at the top of pages by default and can be turned off via `breadcrumbs: false` in the docs preset ([Docusaurus sidebar docs](https://docusaurus.io/docs/sidebar)).

- **Sidebar grouping:** filesystem with `_category_.json` overrides (label, position, link target).
- **Expand/collapse default:** categories collapsed by default; `autoCollapseCategories` enforces a single-open-section discipline.
- **Breadcrumbs:** on by default, derived from the sidebar position (NOT from the URL). This is meaningfully different from GitHub: the same page can render different breadcrumbs in different sidebars.
- **Adopt:** `_category_`-style metadata files are an excellent pattern for our exposed tree if we ever need labels different from folder names.

## 5. GitBook web reader

GitBook recently rebuilt its sidebar with a hover-to-reveal model: the sidebar appears when the user hovers near the left edge and disappears otherwise, yielding "30% more content space when the sidebar is collapsed." The width is adjustable, and sidebar state (expanded/collapsed, open sections) persists via Recoil + localStorage ([GitBook blog: rebuilding the sidebar](https://www.gitbook.com/blog/new-sidebar)). The product team explicitly "optimized for the 90% case—organizations with one to two sites and a handful of spaces."

GitBook recommends **avoiding more than three levels of nesting** "to avoid an overly-complex navigation," and slugs auto-track titles unless manually pinned. Empty parent pages with subpages auto-generate a "contents" landing page ([GitBook page docs](https://gitbook.com/docs/creating-content/content-structure/page)). Breadcrumbs at the site level link back to the main Docs sites screen.

- **Sidebar grouping:** hand-authored TOC with collections > spaces > pages.
- **Expand/collapse default:** state is persisted per user via localStorage.
- **Adopt:** the **3-level nesting cap** is a hard, opinionated UX rule worth borrowing — our tree (project > stage folder > file) already fits.
- **Adopt:** **persist sidebar expand/collapse state in localStorage** so re-opening a project picks up where the user left off.
- **Skip:** hover-to-reveal is gimmicky for a localhost dev tool where the sidebar IS the navigation; a fixed sidebar is more honest.

## 6. Backstage TechDocs

TechDocs is a docs-as-code addon to Backstage that "leverages MkDocs as its underlying renderer." Its layout splits into a `Header`, `Subheader`, `PrimarySidebar` (left, above navigation), main `Content`, and `SecondarySidebar` (right, above the table of contents). Addons declare a `location` property to dock into one of these regions ([Backstage TechDocs Addons](https://backstage.io/docs/features/techdocs/addons/)).

- **Sidebar grouping:** delegated to MkDocs `nav:` (since the renderer is MkDocs Material).
- **Two sidebars:** left = section navigation (where am I in the doc set), right = page TOC (where am I in this page).
- **Adopt:** the **left = tree, right = page TOC** split is the most ergonomic 3-pane reading layout in the survey. For markdown-heavy artifacts, a right-side `## headings` outline is genuinely useful.

## 7. Storybook docs panel

Storybook builds its sidebar from the `title` field of each component file, using `/` as the group separator (e.g. `Foo/Bar` with story name `Baz` produces ID `foo-bar--baz`, accessible at `?path=/story/foo-bar--baz`) ([Storybook sidebar & URLs](https://storybook.js.org/docs/configure/user-interface/sidebar-and-urls)). The hierarchy levels are Category → Folder → Component → Docs/Story. F6 / Shift+F6 cycle keyboard focus between sidebar, toolbar, preview, and addons panel.

Configurable behaviors ([Storybook features and behavior](https://storybook.js.org/docs/configure/user-interface/features-and-behavior)):

- `sidebar.showRoots` (default `false`) controls whether top-level nodes render as visual "roots" (section headers) versus regular folders.
- `sidebar.collapsedRoots: ['misc', 'other']` collapses specific root nodes by default; otherwise nothing is collapsed.
- `sidebar.renderLabel` is a function returning a `ReactNode`, allowing custom label rendering (e.g., `startCase` formatting).
- `showSidebar`, `showPanel`, `showToolbar` are functions of state, allowing per-page hiding (with the docs noting "ensure the displayed page provides an alternative means of navigation" when hiding the sidebar).

- **Adopt:** opt-in **collapsedRoots** pattern — by default expand everything, but allow specific noisy folders (e.g., `findings/`) to default-collapse.
- **Adopt:** **stable IDs decoupled from labels** so renaming a stage folder does not break a saved deep link. (Our paths are filesystem paths, so this is partly a non-issue, but the principle: a deep link should survive a label change.)

## 8. Obsidian Publish

Obsidian Publish reuses Obsidian's two-sidebar model: "navigation is always in the left sidebar and graph in the right sidebar" ([Obsidian forum: Publish layout request](https://forum.obsidian.md/t/publish-move-navigation-graph-and-search-to-same-sidebar-when-outline-is-not-used/25767)). The left sidebar typically holds the file explorer; the right typically holds the outline view, which is "constructed from the structure of the document and understands the structure based on the level of the headings in the document," and clicking an outline item jumps to that location.

- **Sidebar grouping:** raw filesystem (vault).
- **Expand/collapse default:** folders collapsed by default in the file explorer.
- **Skip:** the **graph view** (network of links between notes) is a fun affordance but irrelevant for a strictly hierarchical artifact tree like spec_driven. Adding it would be a vanity feature.
- **Skip:** the rigid "always left + right" sidebar layout has been a pain point for Publish users (the linked forum post is exactly that complaint); a single resizable left sidebar with optional right-side TOC is more flexible.

## Cross-cutting observations

- **Two-pane vs three-pane.** Mature reading tools converge on **three panes**: tree (left), content (center), page TOC (right). MkDocs Material, Docusaurus, Backstage TechDocs, GitBook, and Obsidian all do this. GitHub and Sourcegraph keep TOC in the file (e.g., markdown rendering) without a dedicated right rail, but their primary use case is code, not prose.
- **Default collapse policy is contested.** Material exposes `navigation.expand` (off by default → collapsed), Docusaurus exposes `autoCollapseCategories` (single-open enforcement), Storybook defaults to fully expanded with opt-in `collapsedRoots`. There is no universal best practice; the tree's depth and breadth dictate the right default.
- **Breadcrumbs split into two camps.** GitHub-style (URL-derived) vs. Docusaurus/Material-style (sidebar-position-derived). For an artifact tree where path == identity, URL-derived is honest. Sidebar-derived only makes sense if the sidebar imposes a different ordering than the filesystem.
- **Deep links must be copyable.** Sourcegraph's explicit **Permalink** action and GitHub's `#L42-L60` selection are the strongest patterns. A shareable URL that pastes back into a chat (or a `regen-prompt`) is non-negotiable for a developer tool.
- **State persistence is cheap and high-value.** GitBook's localStorage-backed sidebar state is a small implementation cost for a real ergonomic win, and aligns with the existing `spec_driven.autonomous_mode.v1` localStorage pattern in our webapp.

## Recommendations for spec_driven (5–8)

1. **ADOPT: three-pane layout — tree (left) + content (center) + page TOC (right).** This is the consensus across MkDocs Material, Docusaurus, Backstage TechDocs, GitBook, and Obsidian Publish. The right-side TOC is auto-generated from markdown `##` headings and is read-only. Reuse Backstage's split: left = where am I in the artifact tree, right = where am I in this artifact.

2. **ADOPT: filesystem-derived breadcrumbs above the content pane, GitHub-style.** Each segment links to the corresponding directory listing. Do NOT invent a sidebar-derived breadcrumb (Docusaurus-style) because for spec_driven the path *is* the identity — there is no editorial reordering to surface.

3. **ADOPT: copyable permalinks with optional `#H-{slug}` anchor for headings.** Borrow from Sourcegraph's explicit Permalink button. The URL pattern should be `/{type}/{name}/{stage}/{file}#H-{heading-slug}`, mirroring how GitHub's `#L42` works for code. Anchors must auto-scroll AND highlight the target heading on load.

4. **ADOPT: localStorage-persisted sidebar expand/collapse state** (GitBook pattern). Key by project so each project remembers its own open sections. This pairs naturally with the existing `spec_driven.autonomous_mode.v1` localStorage convention.

5. **ADOPT: opinionated default expand policy — auto-expand the ancestors of the currently selected file, default-collapse everything else.** This is GitHub's behavior, and it scales: with five stage subfolders per project, an aggressive auto-expand (Material's `navigation.expand`) would dump dozens of leaves on the user. Storybook's `collapsedRoots` is a useful escape hatch if `findings/` becomes too noisy — default-collapse `findings/` specifically.

6. **ADOPT: 3-level nesting cap as a design rule** (GitBook's recommendation). Our exposed tree is naturally `project > stage > file`, which fits. Resist adding subfolders beneath stage folders unless absolutely necessary.

7. **SKIP: graph view, hover-to-reveal sidebar, symbol navigation.** Obsidian Publish's graph adds zero value over a hierarchical tree; GitBook's hover-to-reveal makes navigation feel hidden in a localhost dev tool; Sourcegraph's symbol navigation is irrelevant for markdown/YAML/JSON. None of these earn the implementation cost.

8. **SKIP: tabs-on-top + sidebar dual nav (Material's `navigation.tabs`).** Two layers of categorical navigation confuses an opinionated, fixed tree. One sidebar is enough.

## Open questions / not researched

- Whether to support search (full-text or fuzzy filename) inside the sidebar; only Storybook's keyboard-driven search and Sourcegraph's repo-wide search were sampled, neither directly applicable to a fixed exposed tree.
- Mobile / narrow-viewport behavior: most tools (Docusaurus `hideable`, GitBook hover-to-reveal) collapse the sidebar entirely on small screens. Whether spec_driven cares about mobile at all is a product decision, not researched here.
- Right-side TOC rendering for non-markdown files (YAML frontmatter, JSON): no surveyed tool addresses this directly. Likely the TOC should hide for non-markdown files.
- Keyboard navigation conventions: only Storybook's F6/Shift+F6 region cycling was sampled. Whether to adopt arrow-key tree navigation (file-explorer-style) is open.
- Performance on very large trees: GitBook explicitly "optimized for the 90% case—organizations with one to two sites and a handful of spaces." spec_driven's tree is comparable in size, so this is unlikely to bite, but pagination / virtualization was not researched.
