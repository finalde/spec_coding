# Angle — prior-art read-only / hybrid viewers

Run id: spec_driven-20260503-145859

## 1. What this angle covers

How mature read-only and hybrid file-viewer SPAs surface a heterogeneous tree of artifacts beside a "render-everything" main pane. We focused on seven named products — GitHub web UI / web editor (github.dev), VS Code for the Web, Obsidian, MkDocs Material, GitBook, Notion (read mode), and CodeSandbox — and pulled patterns for: (a) sidebar tree affordances, (b) breadcrumb / current-path indication, (c) per-file-type render dispatch, (d) keyboard navigation, (e) deep-link / bookmarkable URLs, and (f) error / fallback rendering when a file can't parse.

The `spec_driven` webapp is essentially the same shape (sidebar tree of `Claude Settings & Shared Context` + `Projects/{type}/{name}/`, render-everything main pane). Convergent practice from these tools is the strongest signal we have for which FRs/NFRs to bake in.

## 2. Key findings

### (a) Sidebar tree affordances

- **GitBook (2024 redesign) explicitly funded a single API endpoint that returns the entire sidebar in one shot, then animates carefully.** They deliberately *avoided* animating expand/collapse "to prevent overwhelming power users" — a quiet vote against jiggle on every node toggle. ([GitBook blog — Rebuilding GitBook's sidebar](https://www.gitbook.com/blog/new-sidebar))
- **MkDocs Material exposes the tree shape as a small, named feature matrix** rather than a single behaviour: `navigation.sections` (top-level groups), `navigation.expand` (auto-expand all collapsibles), `navigation.indexes` (a section node IS a page via `index.md`), `navigation.tabs` (top-level → tabs in the header), `navigation.tracking` (URL hash follows the active anchor). Each is opt-in. ([Material for MkDocs — Setting up navigation](https://squidfunk.github.io/mkdocs-material/setup/setting-up-navigation/))
- **Obsidian's default file explorer is keyboard-hostile enough** that the community ships at least three replacements (Notebook Navigator, Sidebar Keyboard Navigation, Explorer Shortcuts) primarily to add j/k/h/l Vim-style traversal. Hotkey requests are among the longest-standing feature threads on the official forum. ([Obsidian Forum — File Explorer: Navigation Hotkeys](https://forum.obsidian.md/t/file-explorer-navigation-hotkeys/23178), [denvolok/obsidian-sidebar-keyboard-navigation](https://github.com/denvolok/obsidian-sidebar-keyboard-navigation), [Notebook Navigator](https://github.com/johansan/notebook-navigator))

### (b) Breadcrumb / current-path indication

- **Breadcrumbs sit above the H1, never inside the sidebar.** Convergent placement: top of content area, below primary nav, above the page title — and the *current* crumb is rendered as plain text with `aria-current="page"`, not a link. ([UX Patterns for Developers — Breadcrumb Pattern](https://uxpatterns.dev/patterns/navigation/breadcrumb))
- **MkDocs Material treats breadcrumbs as a small-screen orientation aid.** The `navigation.path` feature renders the trail "above the title of each page, which might make orientation easier for users visiting your documentation on devices with smaller screens." It's experimental and opt-in. ([MkDocs Material — navigation.path](https://squidfunk.github.io/mkdocs-material/setup/setting-up-navigation/))
- **Notion's breadcrumb is interactive at every segment.** Click any ancestor in the trail and a dropdown opens listing all *siblings* at that level. The horizontal trail therefore doubles as a lateral-navigation tool, not just a back-button stack. ([Notion Help — Pages, breadcrumbs and sidebar](https://www.notion.so/help/notion-academy/lesson/pages-breadcrumbs-and-sidebar))
- **GitBook keeps a linear breadcrumb back to the docs-site root** even on deeply nested pages, recommending that authors not nest more than three levels deep — past that, breadcrumbs lose orientation value. ([GitBook — Pages](https://gitbook.com/docs/creating-content/content-structure/page))

### (c) Per-file-type render dispatch

- **GitHub's web UI dispatches by extension and content-class:** Markdown → HTML render, code → syntax-highlighted code view with a "Source" / "Preview" toggle for `.md` and similar dual-mode files. Their Sept 2025 changelog adds a "View on default branch" button next to the branch picker for files that exist on default — file-type-aware affordances continue to accrete. ([GitHub Changelog — Improved file navigation and editing in the web UI (2025-09-04)](https://github.blog/changelog/2025-09-04-improved-file-navigation-and-editing-in-the-web-ui/))
- **VS Code for the Web (github.dev)** renders any file the desktop editor can render, dispatching by language id from the file extension; binary or unsupported types fall back to a "binary file not supported" placeholder rather than a crash. The `.` keypress on any github.com page swaps the URL host to `github.dev` and loads the same tree. ([Visual Studio Code for the Web docs](https://code.visualstudio.com/docs/setup/vscode-web), [github.dev web-based editor docs](https://docs.github.com/en/codespaces/the-githubdev-web-based-editor))
- **MkDocs Material treats `index.md` / `README.md` as *the* page for a folder node** (`navigation.indexes`). This is the convergent pattern across MkDocs proper and many others — folder-as-page is the natural read for users. ([Material for MkDocs — navigation.indexes](https://squidfunk.github.io/mkdocs-material/setup/setting-up-navigation/))

### (d) Keyboard navigation

- **VS Code (and github.dev) use `Cmd/Ctrl+Shift+E` to focus the explorer**, then arrow keys for traversal and Enter to open. Long-running issues (microsoft/vscode#18412) show the surprising delta — just *focusing* the sidebar is a chord, but *opening* a focused file isn't always a single keystroke. ([microsoft/vscode#18412 — Keybinding for opening files when navigating sidebar](https://github.com/microsoft/vscode/issues/18412))
- **Obsidian provides `Files: Show file explorer` + `Files: Reveal current file in navigation` as commands** but no built-in j/k traversal — community plugins fill the gap with NERDTree-style bindings. The signal: *power users want vim-style nav out of the box*; default left/right toggling is not enough. ([Obsidian — Sidebar Keyboard Navigation plugin](https://github.com/denvolok/obsidian-sidebar-keyboard-navigation))

### (e) Deep-link / bookmarkable URLs

- **MkDocs Material's `navigation.tracking` writes the active section's anchor into the URL bar as the user scrolls.** The page is therefore bookmarkable not just at the file level but at the heading level — the URL is a live cursor. ([Material for MkDocs — navigation.tracking](https://squidfunk.github.io/mkdocs-material/setup/setting-up-navigation/))
- **Notion goes one step finer: every block has its own anchor.** The block menu's `Copy link to block` produces a URL that re-opens the page scrolled to that exact block. Heading-level deep-links are the floor; block-level is the ceiling. ([Notion Help — Links & backlinks](https://www.notion.com/help/create-links-and-backlinks))
- **github.dev is a pure host swap on the same URL path** — `github.com/<org>/<repo>/blob/<ref>/<path>` ↔ `github.dev/<org>/<repo>/blob/<ref>/<path>`. The deep link survives the host change, which is the simplest possible bookmarkability contract. ([github.dev docs](https://docs.github.com/en/codespaces/the-githubdev-web-based-editor))

### (f) Error / fallback rendering when a file can't parse

- **YAML frontmatter is the load-bearing failure point in markdown viewers.** Documented failures span CRLF line endings, special characters in unquoted fields, and overly strict parsers crashing on otherwise-valid content. Mature viewers (Foam PR #320, R Markdown) have moved toward parser-friendly degradation rather than hard crash. ([foambubble/foam#320 — improved error handling when parsing markdown files](https://github.com/foambubble/foam/pull/320), [github/copilot-cli#694 — Agents fail to load if frontmatter has CRLF line endings](https://github.com/github/copilot-cli/issues/694), [anomalyco/opencode#8331 — YAML frontmatter parser too strict](https://github.com/anomalyco/opencode/issues/8331))
- **The convergent fallback shape is "render the raw text + a banner"**, not a blank screen and not an alert dialog. R Markdown's `15 Common Problems` guide and Inkdrop's "Failed to render Markdown" thread both push users toward "show the file as plain text and surface the parse error inline so the user can fix it." ([RMarkdown for Scientists — common problems](https://rmd4sci.njtierney.com/common-problems-with-rmarkdown-and-some-solutions.html), [Inkdrop forum — Failed to render Markdown](https://forum.inkdrop.app/t/failed-to-render-markdown-after-import-from-markdown/2039))
- **Binary / unsupported types get a placeholder pane with the file's metadata,** not a 404 or a crash — VS Code Web's "binary file not supported" stub is the pattern. ([VS Code for the Web docs](https://code.visualstudio.com/docs/setup/vscode-web))

## 3. Implications for the spec (concrete, actionable)

These map to FR/NFR/AC slices already structured in the spec_driven pipeline; I've tried to call out which.

1. **Tree node uniformity.** `GET /api/tree` already returns recursive `{name, path, type, children: []}` (per qa.md, FR-7/AC-3). Lock that in — every named viewer above sends a single recursive payload, never two heterogeneous shapes. Drift between `task_type.projects` / `project.stages` and `node.children` is exactly the GitBook + MkDocs anti-pattern they engineered around.
2. **Default-expand state for top-level sections, lazy for nested.** Mirror MkDocs Material's `navigation.expand`: the two top-level sections (`Claude Settings & Shared Context`, `Projects`) and each project's stage subfolders should be expanded by default; deeper folders collapse by default. Power users do not want to click 5 disclosure triangles to find `final_specs/spec.md`. (Affects FR-7 / FR-8.)
3. **Breadcrumb above the main pane, not inside the sidebar; current crumb is text with `aria-current="page"`.** Match GitBook + MkDocs + Notion convergence. The breadcrumb segments should be clickable links to ancestor folders (or to the project root). Notion-style sibling-dropdown is *nice-to-have* and out-of-scope for v1; explicitly note that. (New FR candidate: breadcrumb component.)
4. **Per-file-type render dispatch by extension, with a `README.md`-as-section-page convention.** `.md` → markdown render with link rewriting (already FR-33..35); `.json` / `.yaml` / `.yml` / `.jsonl` / `.txt` → syntax-highlighted code view (already FR-3 in qa.md); `.png` / `.jpg` → placeholder block; everything else → 415 (already FR-3). Add: when a folder node's child set contains `README.md`, *clicking the folder* in the sidebar opens `README.md` in the main pane rather than just toggling the disclosure. This matches MkDocs `navigation.indexes` and GitBook subpage behaviour. (Refines FR-7 / FR-8.)
5. **Keyboard nav baseline.** At minimum: `Up` / `Down` move selection within the visible tree, `Right` expands a folder, `Left` collapses it, `Enter` opens the selected file. This is the VS Code default and the floor every Obsidian alternative-explorer plugin replicates. Vim-style j/k is out-of-scope for v1 but call it out in `## Out of scope` so we don't drift. (New FR candidate: keyboard navigation contract.)
6. **Deep-link contract.** A pasted URL `/file?path=specs/development/spec_driven/final_specs/spec.md` MUST (a) render the file, (b) expand all ancestor folders in the sidebar, (c) highlight the file node, (d) update the breadcrumb. github.dev's host-swap pattern proves the floor: the URL is the canonical state. Heading-level anchors (`#fr-7`) are nice-to-have; block-level (Notion) is out-of-scope. (Refines FR-3 / new AC candidate.)
7. **Parse-failure fallback is "raw text + banner", never blank + alert.** When markdown frontmatter or `interview/qa.md` structured parse fails, render the raw file contents in a `<pre>` with a banner explaining the failure (already in qa.md as `QaErrorBoundary` move 9 / AC-25). Generalize that contract to all renderers: a parse failure is always a soft fallback to plain-text view, never a 5xx, never a blank pane. (Generalize AC-25 across FR-33..41.)
8. **Single-shot tree fetch.** GitBook explicitly funded one endpoint returning the whole sidebar to kill chattiness. `GET /api/tree` already does this; resist the urge to add per-folder lazy endpoints during stage 6 — the workflow's tree is bounded enough that one shot is correct.

## 4. Open questions surfaced

1. **README-as-section-page** — should clicking a folder open its `README.md` if present, or only render the folder placeholder? MkDocs makes it opt-in (`navigation.indexes`); we could either default it on (less click cost) or off (more predictable). Worth a follow-up if user signals either way.
2. **Breadcrumb sibling-dropdown** — Notion's pattern (click any crumb → dropdown of siblings) is genuinely useful for a deeply nested workflow tree. Spec it as v1, defer to v2, or skip entirely?
3. **Heading-level deep links inside markdown** — MkDocs Material auto-tracks the active heading in the URL bar (`navigation.tracking`). Should `spec_driven` do the same so a link to `#fr-7` survives a paste? Adds React-router complexity; may be out-of-scope for v1.
4. **Vim-style keyboard nav** — is the population of users who'd use j/k traversal large enough to justify the additional keymap? The Obsidian community signal is strong but this app is single-user and short-running — likely defer.
5. **Empty / orphaned tree nodes** — what does the sidebar render for a `task_type` folder that exists on disk but is empty (no `task_name` subfolders yet)? Show as empty disclosure, hide entirely, or render an "empty state" tile? Convergent practice (MkDocs, GitBook) is *hide entirely*; check if that conflicts with the discoverability goal of the `Projects/` section.
