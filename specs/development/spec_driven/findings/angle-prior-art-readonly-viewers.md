# Angle — Prior art for readonly artifact viewers

## 1. What this angle covers

This angle surveys established products that solve a structurally similar problem to the `spec_driven` viewer: rendering a hierarchical tree of generated, mostly-markdown artifacts in a browser, with a left sidebar for navigation and a main pane for content. The goal is to extract recurring UX patterns, named anti-patterns, and concrete moves to adopt or skip given the locked v1 profile (readonly, single-process, ≤50 projects, Chrome only, dogfood-first).

Probes covered: GitHub web UI, Sourcegraph, MkDocs Material, Docusaurus, GitBook, Obsidian Publish, Backstage TechDocs, Storybook, FileGator/h5ai/Dufs, VS Code Explorer (as the closest "tree of files" interaction model), and general docs-site UX literature (Smashing/NN/g).

## 2. Key findings

### 2a. Recurring patterns across 3+ tools

| Pattern | Where it shows up | Note |
|---|---|---|
| **Sidebar grouped by category, collapsible per section** | MkDocs Material (`navigation.sections`), Docusaurus (autogenerated categories with `collapsed`/`collapsible`), Storybook ("roots" rendered as sections, lower nodes as folders), GitBook | Universal pattern for nested doc hierarchies. ([MkDocs Material navigation](https://squidfunk.github.io/mkdocs-material/setup/setting-up-navigation/), [Docusaurus autogenerated](https://docusaurus.io/docs/next/sidebar/autogenerated), [Storybook sidebar](https://storybook.js.org/docs/configure/user-interface/sidebar-and-urls)) |
| **Auto-expand path to active page; collapse the rest** | MkDocs Material default behaviour, Docusaurus (`collapsed: false` for the active branch), GitBook | "The navigation will always expand the path to the active page and collapse all non-active pages." ([MkDocs Material](https://squidfunk.github.io/mkdocs-material/setup/setting-up-navigation/)) |
| **Breadcrumbs above the title for orientation in deep trees** | MkDocs Material, GitHub repo browser, Sourcegraph file view | Treated as a *supplement* to the sidebar, not a replacement ([NN/g breadcrumbs guidelines](https://www.nngroup.com/articles/breadcrumbs/), [Smashing breadcrumbs](https://www.smashingmagazine.com/2022/04/breadcrumbs-ux-design/)) |
| **Persistent active-item highlight** | All doc sites (Material, Docusaurus, GitBook), Storybook | Highlighted state in the sidebar communicates "you are here" — recommended as an alternative to breadcrumbs when hierarchy is shallow ([Smart Interface breadcrumbs](https://smart-interface-design-patterns.com/articles/breadcrumbs-ux/)) |
| **Single-pane render of selected file (markdown → HTML; code → tokenised)** | GitHub markup pipeline, Sourcegraph, MkDocs, Docusaurus, Obsidian Publish | GitHub's own pipeline does syntax highlighting on code blocks and adds emoji/task lists/autolinks ([github/markup](https://github.com/github/markup)) |
| **Built-in/static-served single artifact (no separate dev/prod servers in published form)** | MkDocs (static), Docusaurus (static), GitBook (rendered out), Backstage TechDocs (built docs served from object store) | Mirrors the locked decision to have one FastAPI process serve the built React bundle ([Backstage TechDocs architecture](https://backstage.io/docs/features/techdocs/architecture/)) |
| **Cross-link resolution between artifacts** | Obsidian Publish (links + backlinks), MkDocs, Docusaurus | Treated as a first-class feature; relative `.md` links rewrite to in-app routes ([Obsidian Publish](https://publish.obsidian.md/)) |
| **Animation when sidebar expands/collapses to communicate spatial continuity** | GitBook redesign, MkDocs Material | "Helps users understand the spatial relationship and makes it clear that the same element is moving, not appearing and disappearing." ([GitBook sidebar rebuild](https://www.gitbook.com/blog/new-sidebar)) |

### 2b. Anti-patterns to avoid

- **Breadcrumb-as-history**: showing the click trail rather than the structural path. NN/g calls this out explicitly — breadcrumbs must reflect site hierarchy, not session order ([NN/g](https://www.nngroup.com/articles/breadcrumbs/)).
- **Last-crumb-as-link**: rendering the current page's breadcrumb as a clickable link creates "confusing circular navigation" ([Smart Interface](https://smart-interface-design-patterns.com/articles/breadcrumbs-ux/)).
- **Tree state lost on every navigation**: VS Code's Explorer tree is the canonical example — folder expand/collapse state is *not persisted across launches*, and Alt+Tab can fully collapse the tree in multi-root workspaces. Multiple long-standing issues exist ([vscode#116190](https://github.com/microsoft/vscode/issues/116190), [vscode#228788](https://github.com/microsoft/vscode/issues/228788)). Users notice and complain.
- **Aggressive auto-reveal**: when the editor switches files and the tree forcibly expands to show the new file, it disrupts users who manually collapsed branches. VS Code added `explorer.autoReveal: false` as an escape hatch ([vscode#42146](https://github.com/microsoft/vscode/issues/42146)).
- **Permanent icon-only rail consuming vertical space**: GitBook explicitly moved away from this in their redesign because it ate "precious vertical space" ([GitBook blog](https://www.gitbook.com/blog/new-sidebar)).
- **Generic FTP-style file managers as the model**: FileGator, h5ai, and Dufs are excellent raw-filesystem browsers, but their UX optimises for *file operations* (upload/download/zip/permissions) not for *reading rendered structured artifacts*. Borrowing their layout would import affordances the readonly viewer must then explicitly suppress ([filegator README](https://github.com/filegator/filegator), [dufs README](https://github.com/sigoden/dufs)).
- **Recursive expand-all on a stage-heavy hierarchy**: Storybook ships this as a power-user shortcut precisely because it's painful by default in large libraries — the spec_driven tree (≤50 projects × 5 stages) is small enough that the feature is unnecessary, but the lesson stands: do not *default* to expanded-everything.

### 2c. UX moves the spec_driven viewer should adopt

1. **Sidebar = three subgroups in section 1, then `task_type` collapsibles in section 2** — this matches the locked qa.md decision and matches the Material/Docusaurus pattern of category-as-collapsible-section.
2. **Auto-expand the path to the currently selected file; collapse the rest by default**. Adopt MkDocs Material's documented default — it is the dominant convention.
3. **Persist sidebar collapse state in `localStorage`** keyed by node path, plus the last-selected file path. This is the single biggest UX win VS Code's Explorer fails at, and `localStorage` makes it ~20 lines of code. Restore on reload.
4. **Breadcrumb strip above the rendered file** showing `task_type / task_name / stage / filename`. Last crumb is plain text, not a link. Other crumbs navigate the tree. Treat as orientation aid, not a replacement for the sidebar.
5. **Cross-link resolution**: relative markdown links that resolve to a file inside `specs/` rewrite to in-app routes; external links open in a new tab; broken links render muted with tooltip. This matches the qa.md "internal clickable" decision and Obsidian Publish's behaviour.
6. **Single landing route → spec_driven's final spec (or revised_prompt fallback)**. Mirrors how Backstage TechDocs lands users on a component's docs page directly from the catalog.
7. **No animation budget beyond a 100–150ms ease on sidebar expand/collapse**. GitBook found this pays off in perceived continuity; anything more is noise for a dogfood tool.
8. **Skip these prior-art features deliberately**:
   - Search (locked out for v1)
   - Recursive expand-all / collapse-all toolbar (no tree depth justifies it)
   - Backlinks / graph view (Obsidian's signature feature; spec_driven's links are mostly forward, stage-to-stage)
   - Per-page metadata sidebar (GitBook/Backstage have it; spec_driven has no metadata to display)

## 3. Implications for the spec

- **Sidebar component** must support: per-node collapse state, persisted to `localStorage`, with the active path auto-expanded on first load. State key suggestion: `spec_driven.sidebar.v1`.
- **Routing** should be path-based: `/settings/claude-md`, `/settings/agents/<file>`, `/settings/skills/<folder>`, `/projects/<task_type>/<task_name>/<stage>/<file>`. This makes deep-links shareable and breadcrumb construction trivial.
- **Markdown renderer** must post-process relative links: resolve against the file's directory; if the resolved path lies inside the exposed tree, rewrite to an in-app route; otherwise tag external/broken accordingly. This is non-trivial and deserves its own work unit.
- **Empty-folder rendering**: a stage subfolder may be missing (e.g. `validation/` doesn't exist yet for early-stage projects). The spec must say what shows: hide the leaf? Show it disabled? Show "not yet generated" placeholder? Docusaurus and MkDocs both surface every entry the build sees — they have no concept of "expected but missing." spec_driven needs to take a position because the *expected* tree shape is fixed.
- **Breadcrumb component** is small enough (~50 lines) that it's worth specifying as a separate work unit rather than folding into the file-viewer.
- **No file watcher / no auto-refresh** is locked, but a manual "reload tree" button (no live updates) is a reasonable affordance and is precedented by every static doc site (which requires a rebuild). Decision left to spec author.

## 4. Open questions surfaced

1. **Empty / missing stage folders** — render the leaf as disabled, hide it, or show "not yet generated"? (Strongly recommend: render it muted/italic with a tooltip — matches the broken-link convention and signals expected-but-empty.)
2. **Sub-stage files of unknown shape** — `validation/` may contain `strategy.md`, `acceptance_criteria.md`, `bdd_scenarios.md`, plus optional non-functional files. Does the sidebar list every `*.md` it finds (alphabetical, like Docusaurus default), or impose an order? Recommend: alphabetical with a small known-name priority list (`strategy.md` first).
3. **Manual reload affordance** — is a "refresh tree" button in scope, or strictly browser reload only?
4. **Clipboard / "copy link" affordance** on rendered files — most doc sites have it for headings; cheap to add but unspecified. Skip for v1?
5. **Section 1 link-resolution scope** — does a markdown link from `CLAUDE.md` to `.claude/agents/foo.md` resolve in-app the same way `specs/`-internal links do? qa.md's "internal clickable" answer is ambiguous about whether it covers section 1 or is `specs/`-only.
6. **Code/JSON/JSONL line-number anchoring** — Shiki supports it; should the URL fragment `#L42` scroll to and highlight a line, like GitHub does? Cheap, very useful when sharing references during a review.
