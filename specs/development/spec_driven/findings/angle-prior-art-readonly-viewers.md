# Angle — prior-art-readonly-viewers

**Run:** spec_driven-20260503-fullregen
**Stage:** 3 (Research)
**Angle owner:** researcher-prior-art-readonly-viewers
**Method:** WebSearch + WebFetch against vendor docs and engineering posts. Read-zero from any prior `findings/*`.

## Framing

`spec_driven` ships a left-nav sidebar and a reader pane over a tree of mostly-markdown artifacts (`CLAUDE.md`, `.claude/agents/*.md`, `.claude/skills/**/SKILL.md`, plus the per-stage tree under `specs/{type}/{name}/`). The interview pinned `pin-001`: enumerate every `specs/{type}/{name}/` on disk — no hardcoded project list — so the sidebar must scale from "one project, six folders" to "many projects, hundreds of files" without a redesign. The reader pane must support deep-links (the regen panel is per-file, so URLs need to round-trip), in-place editing, and a "missing artifact" empty state. Eight mature tools already solved variants of this problem: GitHub's web UI, Sourcegraph, MkDocs Material, Docusaurus, GitBook, Backstage TechDocs, Storybook, and Obsidian Publish. This angle compares them on five axes — sidebar grouping conventions, auto-expand vs collapse-by-default, breadcrumbs, deep-link behavior, anti-patterns to skip — and ends with 5–8 ADOPT/SKIP recommendations for spec_driven's sidebar + reader pattern.

## Per-tool comparison

### 1. Docusaurus

**Grouping conventions.** Two primitive types: `category` (a hierarchical folder-like group) and `link` (a leaf, internal page or external URL). Sidebars can be hand-authored in `sidebars.js` or auto-generated from the filesystem with `type: 'autogenerated', dirName: '.'`. Per-folder metadata files (`_category_.json`) supply display names and ordering.

**Auto-expand vs collapse.** Categories are expandable/collapsible by default. The high-leverage option is `themeConfig.docs.sidebar.autoCollapseCategories`: "would collapse all sibling categories when expanding one category" — i.e., accordion-style, only one branch open at a time. This is opt-in, not the default.

**Breadcrumbs.** Rendered by default at the top of each doc. Disable globally via the plugin option `breadcrumbs: false`.

**Deep-link behavior.** Docusaurus is a SPA-with-MDX; links are real URLs that match the filesystem path used during sidebar generation. The sidebar tree expands the path to the active doc on navigation.

**Anti-patterns to skip.** Open GitHub issues complain that "sidebar categories don't autocollapse correctly" in some configurations and that label-vs-arrow-only collapse is a long-standing UX papercut (issue #10854). Folks also report autogenerated-sidebar limitations (#5689) — meaning a fully filesystem-driven nav is fragile when the tree has irregularly-named or unconventional files. Lesson: don't try to derive every label and ordering from the filename alone; let the backend supply display metadata.

### 2. MkDocs Material

**Grouping conventions.** Driven by `mkdocs.yml`'s `nav:` block (or auto-derived from the filesystem if omitted). Top-level entries can render as sidebar groups (`navigation.sections`) and/or as a tabs strip across the top (`navigation.tabs`). The two compose: when both are on, sections render at level 2.

**Auto-expand vs collapse.** Default behavior, quoted from the maintainer: *"The navigation will always expand the path to the active page and collapse all non-active pages."* Opt-in `navigation.expand` expands every collapsible subsection on every page. Critically, `navigation.expand` is **incompatible** with `navigation.prune` (a perf optimization that ships only the active subtree).

**Breadcrumbs.** `navigation.path` (experimental) renders breadcrumbs above the page title. The maintainers position this as a small-screen aid rather than a desktop default.

**Deep-link behavior.** `navigation.tracking` updates the URL hash automatically as TOC anchors scroll into view; `toc.follow` keeps the active anchor visible in the right rail. `navigation.instant` upgrades internal-link clicks to XHR fetches, which keeps sidebar state alive across navigations — the maintainer notes that without instant-loading, "this is not a single-page application, so the state is generally lost/replaced" (acknowledging that preserving expand/collapse state across full-page reloads is "not trivial to implement").

**Anti-patterns to skip.** `navigation.expand` everywhere is loud on big trees and forces full-tree shipping (incompatible with pruning). Persisted expand/collapse state is a hard problem if the app is multi-page; spec_driven is already SPA so this is free for us — but only if the sidebar tree component is mounted once at app shell level, not re-mounted per route.

### 3. Backstage TechDocs

**Grouping conventions.** TechDocs renders MkDocs output, so the sidebar **inherits MkDocs Material conventions** — quoting the official docs: *"Backstage uses an opinionated theme based on material-mkdocs."* `mkdocs.yml`'s `nav:` defines structure; without it, Backstage mirrors the filesystem.

**Auto-expand vs collapse.** Same defaults as MkDocs Material (active-path expansion, siblings collapsed).

**Breadcrumbs.** TechDocs adds a header strip with the entity context (component/system/owner) above the MkDocs page; in-page breadcrumbs are MkDocs-driven.

**Deep-link behavior.** Backstage's URL scheme nests the docs site under the entity ref (`/docs/<kind>/<namespace>/<name>/...`). Inside the doc tree, anchor following uses the underlying MkDocs behavior. There are known issues — e.g., Backstage GitHub issue #19324, "techdocs navigation bar & toc not hiding" — that signal the embed-an-MkDocs-build approach has rough edges around when to show/hide the sidebar.

**Anti-patterns to skip.** Coupling a sidebar to a baked static-site output (the way TechDocs renders pre-built MkDocs) loses live-edit affordances. spec_driven explicitly needs editing in place, so we should not adopt the "build artifact, serve static" pipeline; we should keep the React shell on top of a live filesystem read, more like Sourcegraph than TechDocs.

### 4. Storybook

**Grouping conventions.** Stories form an implicit tree from `title` strings using `/` as the path separator. The recommended pattern, quoting the docs, is to *"mirror your filesystem structure"* — `components/modals/Alert.stories.js` with `title: 'Components/Modals/Alert'`. Top-level nodes display as "roots"; deeper levels as folders. `sidebar.showRoots: false` flattens the root layer.

**Auto-expand vs collapse.** The sidebar's expand/collapse state is **not customizable out-of-the-box**; that's a long-standing complaint (issue #244, "Expand all categories" — open since the early Storybook days). The community add-on `@a110/storybook-expand-all` exists specifically to expand everything on launch.

**Breadcrumbs.** None as a header element. The active path is communicated by the sidebar's expanded state and the URL ID.

**Deep-link behavior.** Stories deep-link via auto-generated IDs derived from `title` + story name (`?path=/story/foo-bar--baz`). Renames break those URLs — Storybook lets you set an explicit `id` on the meta to keep permalinks stable across refactors.

**Anti-patterns to skip.** Implicit IDs from title-paths means rename-breakage is the default; engineers have to remember to pin `id` on rename. spec_driven's URL scheme should derive deep-links from the on-disk path directly so a rename is a rename, not a permalink break — but the pin-001 enumerate-everything contract already forces this (the URL is the path).

### 5. GitBook

**Grouping conventions.** Pages nest into spaces; spaces nest into a sidebar that "holds all of your documentation, as well as notifications, the search bar, snippets and more." GitBook recommends limiting nesting: *"there's no limit to page nesting, but it's recommended to avoid more than three levels of nesting to avoid an overly-complex navigation."*

**Auto-expand vs collapse.** GitBook's 2024 sidebar redesign blog explicitly avoided animating individual expand/collapse: *"expanding and collapsing individual sections within the tree doesn't use animation. It would be overwhelming and slow down power users."* They persist open/collapsed preferences across sessions via Recoil atoms backed by localStorage.

**Breadcrumbs.** Headings auto-generate anchor links. *"You can see anchor links in public content … by hovering over the title and clicking the # that appears next to it, which will update the URL in your browser's top bar so you can copy it to use elsewhere."* Page-level breadcrumbs aren't the centerpiece; the sidebar tree is the orienting affordance.

**Deep-link behavior.** Headings → anchors → shareable URLs. Standard.

**Anti-patterns to skip.** GitBook's redesign post is candid: animation for its own sake hurts repeat users. *"Animations should help users understand what's happening, show progress, or bring joy without compromising function."* Keep tree expand/collapse instant, no fade/slide.

### 6. Sourcegraph

**Grouping conventions.** File-tree-on-the-left mirrors the repo's filesystem exactly. The right rail hosts a separate symbol sidebar, which "categorizes declarations by type (variable, function, interface, etc)" — orthogonal to the file tree.

**Auto-expand vs collapse.** Sourcegraph auto-expands the path to the currently viewed file; sibling subtrees stay collapsed. Keyboard-first UX: Alt+T toggles the tree; arrows navigate.

**Breadcrumbs.** A path breadcrumb sits above the file viewer. There's a known UX issue (sourcegraph/sourcegraph #24946) flagging that the root-directory link in the breadcrumbs UI was missing — i.e., breadcrumbs need a clickable root segment. Worth replicating that affordance correctly from day one.

**Deep-link behavior.** URLs encode `<host>/<owner>/<repo>@<rev>/-/blob/<path>` plus an optional line range. The Permalink action explicitly copies a URL to the *exact line range* you're viewing. For spec_driven, the analog is "URL encodes path + optional anchor for a specific qa.md round/category."

**Anti-patterns to skip.** Symbol sidebar + file sidebar simultaneously is information-dense and only justified for source code. spec_driven's artifacts are mostly-prose markdown; a single tree on the left is enough.

### 7. GitHub web file browser

**Grouping conventions.** A single tree view, optionally toggled open as a left rail (rolled out broadly in 2022–2023). Each node is one filesystem entry. Folders are alphabetical. Breadcrumbs sit in the file header (`<owner>/<repo>/<branch>/<path>` segments).

**Auto-expand vs collapse.** The tree expands the path to the active file and collapses siblings. State does not persist across hard navigations between repos.

**Breadcrumbs.** First-class. Each path segment is a clickable, navigable node; the branch picker lives at the breadcrumb level. A 2025 changelog entry adds a "View on default branch" button next to the branch picker — addressing the friction of navigating away from a non-default branch view of a file.

**Deep-link behavior.** URL is the source of truth: `/<owner>/<repo>/blob/<branch>/<path>#L<n>-L<m>`. Editing pivots off that URL: the September 2025 update lets users edit "if the user is on the latest commit of the file on the default branch" via an inline "Edit file…" dropdown — a model spec_driven's editor toggle echoes.

**Anti-patterns to skip.** GitHub historically lacked any tree at all on file pages — Sourcegraph's "file tree navigation on GitHub? Yes, please" blog and the Gitako extension exist because users wanted exactly this. Lesson: don't ship a viewer without a persistent left-tree from day one. spec_driven already plans this; treat it as table-stakes.

### 8. Obsidian Publish

**Grouping conventions.** Sidebar mirrors the vault's folder structure verbatim. No grouping primitive beyond folders.

**Auto-expand vs collapse.** Folders collapse by default; clicking expands. Sort is **alphabetical only** with no native manual override. Forum thread (Obsidian forum #12041) summarizes the constraint: *"The file explorer is organized alphabetically with no way of changing the order"* and *"there is no way to put a file above a folder or mix those together in any way."*

**Breadcrumbs.** Minimal; the publish site emphasizes the note title and inline links.

**Deep-link behavior.** URLs match the published note path; standard.

**Anti-patterns to skip.** This is the cautionary tale. Letting the filesystem alone dictate sidebar order means users can't promote a homepage, can't make `intake` come before `interview` (alphabetical would put `final_specs` first), and have to resort to numeric prefixes like `01-…`. spec_driven MUST supply a stage ordering hint — the natural pipeline order is `user_input → interview → findings → final_specs → validation` — independent of alphabetical sort.

## Recommendations for spec_driven (ADOPT / SKIP)

1. **ADOPT — active-path-only expand, siblings collapsed (MkDocs/Sourcegraph/GitHub default).** *"The navigation will always expand the path to the active page and collapse all non-active pages"* is the dominant convention across MkDocs, Sourcegraph, and GitHub; users expect it. Implement it as: on route change, expand the chain from root to the selected file; do not auto-collapse user-opened sibling branches (don't be aggressive). SKIP Docusaurus's `autoCollapseCategories` accordion mode — the user wants to compare files across stages (e.g., `qa.md` and `spec.md` open simultaneously in two tabs), so accordion-collapsing siblings on every click is hostile.

2. **ADOPT — explicit stage ordering, not alphabetical (counter to Obsidian Publish).** Hardcode the stage order `user_input, interview, findings, final_specs, validation` in the backend tree-walker response. Inside each stage, alphabetical is fine for now. This avoids Obsidian Publish's "01-prefix" workaround. The order lives in the FastAPI tree endpoint, not the frontend, so it's a single source of truth.

3. **ADOPT — deep-links are the file path, no separate ID scheme (Sourcegraph/GitHub model, contra Storybook).** URL = `/file?path=<urlencoded-path>` (or `/file/<path>` if cleaner). A rename of `qa.md` is a permalink break by design — it's the same break git would record. Don't build a Storybook-style decoupled `id`. For the structured Q&A view, support a `#round-N-category-X` anchor that the parser resolves to the corresponding block; this lets the regen-panel deep-link to a specific Q&A pair if needed later.

4. **ADOPT — clickable, full-path breadcrumbs (Sourcegraph/GitHub), each segment a real link.** Render `Projects / development / spec_driven / interview / qa.md` above the reader pane. Each segment navigates to that subtree (clicking `interview` selects the folder, expanding it in the sidebar; clicking `spec_driven` selects the project parent page at `/project/development/spec_driven`). Explicitly include the **root segment** — Sourcegraph issue #24946 was filed because that segment was missing. Cheap to get right; expensive to retrofit.

5. **ADOPT — single tree on the left, no symbol/secondary sidebar (contra Sourcegraph dual-sidebar).** The artifacts are prose markdown plus a few YAML/JSON files. No symbol indexing pays off here. Keep one sidebar. Reserve the right rail for the per-stage Regenerate panel and editor controls.

6. **ADOPT — instant tree expand/collapse, no animation (GitBook's explicit lesson).** Quoting the GitBook redesign post: *"Expanding and collapsing individual sections within the tree doesn't use animation. It would be overwhelming and slow down power users."* spec_driven's user is the workflow's owner — definitionally a power user. Use CSS `display: none/block` toggles, not framer-motion. Persist the expanded set in `localStorage` under `spec_driven.sidebar.expanded.v1` so it survives reloads (GitBook's Recoil-atoms-to-localStorage approach, simpler).

7. **ADOPT — missing-artifact empty state mounts the regen panel (qa.md-confirmed in stage 2).** Already locked in by the qa.md "Empty state + regen panel mounted" answer; reinforced by GitHub's 2025 "View on default branch" pattern of putting the next-step CTA *in the file pane itself*. Render `Not yet generated — paste this prompt to produce it` with the Regenerate panel mounted directly below, so the user never has to navigate elsewhere to generate a missing file.

8. **SKIP — building a static-site pipeline (Backstage TechDocs / GitBook publish) instead of a live filesystem reader.** TechDocs/GitBook invest heavily in a build step that bakes markdown into HTML. spec_driven needs in-place editing, missing-artifact regen affordances, and re-render-on-save — all of which a live `GET /api/file` + client-side markdown render handles trivially. The build-and-serve detour adds zero value and breaks edit semantics. Render markdown client-side with a library (e.g., `react-markdown` + `prism` for code blocks); YAML/JSON show as monospace highlighted text per the qa.md decision.

## Open questions / not researched

- **Accessibility of the tree component.** ARIA `tree`/`treeitem` semantics, keyboard navigation (arrow keys, Home/End, Type-ahead) — not investigated this round; should be a stage-5 a11y concern referenced from this angle.
- **Search across artifacts.** Explicitly out of scope per revised_prompt; not researched.
- **Watching for file changes (auto-refresh).** Out of scope per revised_prompt; not researched.
- **Persistence of expand/collapse state across users / multi-user contention.** Localhost-single-user model makes this a non-issue today; if the security model ever changes, revisit.
- **Mobile/responsive sidebar collapse.** MkDocs Material has explicit small-screen breadcrumb logic; spec_driven's localhost-desktop bias means this is deferred.
- **Markdown extensions in qa.md** — the structured-Q&A parser uses regex on the raw markdown; whether downstream Markdown-It / remark plugins (admonitions, tables in answers) interfere with the round-trip parse is a stage-6 implementation concern, not a prior-art question.
