# Angle: localhost-fs-sandbox-risks

## 1. What this angle covers

The spec_driven webapp is a FastAPI + React application bound to `127.0.0.1:8765` that exposes a fixed `EXPOSED_TREE` of the developer's repository (CLAUDE.md, `.claude/`, `specs/`, etc.) for read **and write** via `PUT /api/file`. Even though it never binds `0.0.0.0`, the user explicitly noted that "localhost is not a security boundary on a multi-user dev box; defense in depth matters." This angle surveys the classes of vulnerability that have repeatedly surfaced in localhost-only file viewers/editors (Vite dev, Next.js dev, MkDocs serve, Jekyll, JupyterLab, code-server-style extensions) and translates the resulting OWASP guidance into concrete, Windows-aware sandboxing rules for `safe_resolve`, the upload pipeline, and the response layer.

## 2. Key findings

### 2a. Localhost dev servers are a recurring CVE source

- **Vite dev server has been path-traversed at least four ways in 2025 alone.** `CVE-2025-30208` lets an unauthenticated attacker read arbitrary files via `/@fs/...` plus `?import&raw??`; `CVE-2025-46565` bypasses `server.fs.deny` with `/.` prefixing; `CVE-2025-58752` mishandles HTML file paths in `indexHtmlMiddleware`; **`CVE-2025-62522` is Windows-specific — `\` (backslash) bypasses `server.fs.deny` because deny matched only forward-slash forms.** ([offsec.com](https://www.offsec.com/blog/cve-2025-30208/), [snyk.io / CVE-2025-46565](https://security.snyk.io/vuln/SNYK-JS-VITE-9919777), [miggo.io / CVE-2025-58752](https://www.miggo.io/vulnerability-database/cve/CVE-2025-58752), [miggo.io / CVE-2025-62522](https://www.miggo.io/vulnerability-database/cve/CVE-2025-62522))
- **MkDocs `mkdocs serve` had directory traversal on port 8000 (`CVE-2021-40978`)** until 1.2.3; Jekyll had the same issue pre-1.4.3. ([github.com/mkdocs](https://github.com/mkdocs/mkdocs/issues/2601), [snyk.io / Jekyll](https://security.snyk.io/vuln/SNYK-RUBY-JEKYLL-20383))
- **JupyterLab and friends:** `jupyterlab-lsp` (`CVE-2024-22415`) allowed path traversal beyond the Jupyter root when filesystems were configured without OS-level access control; nbconvert's markdown renderer (`CVE-2026-39378`) reads arbitrary files via `../`-laden image references when `embed_images` is on. ([snyk.io / CVE-2024-22415](https://security.snyk.io/vuln/SNYK-PYTHON-JUPYTERLABLSP-6180562), [hackthebox.com](https://www.hackthebox.com/blog/path-traversal-vulnerabilities-in-jupyter-nbconvert))
- **VS Code extensions hosting localhost HTTP servers** (LaTeX Workshop ~1.2M installs, "Open in Default Browser") have repeatedly leaked arbitrary local files via path traversal combined with SSRF / `Window.postMessage`, demonstrating that "localhost-only" is not a meaningful boundary when a browser tab can reach the loopback. ([snyk.io / VS Code extensions](https://snyk.io/blog/vulnerable-visual-studio-code-extensions-marketplace/), [trailofbits.com](https://blog.trailofbits.com/2023/02/21/vscode-extension-escape-vulnerability/))
- **DNS rebinding turns any malicious tab into a localhost client.** A short-TTL DNS record can rebind an attacker domain to `127.0.0.1`, defeating same-origin protections. The OWASP-blessed application-layer mitigations are TLS, authentication, and **HTTP `Host`/`Origin` validation** at the server. ([Stanford paper](https://crypto.stanford.edu/dns/dns-rebinding.pdf), [rafter.so](https://rafter.so/blog/mcp-dns-rebinding-localhost))

### 2b. OWASP path-sandboxing guidance, with Windows specifics

| Concern | OWASP / vendor guidance | Windows wrinkle |
|---|---|---|
| Traversal sequences | Reject `..`, normalize then verify the resolved path is within the allowed root. ([OWASP Path Traversal](https://owasp.org/www-community/attacks/Path_Traversal), [OWASP WSTG 4.5.1](https://owasp.org/www-project-web-security-testing-guide/latest/4-Web_Application_Security_Testing/05-Authorization_Testing/01-Testing_Directory_Traversal_File_Include)) | Backslash `\` is an alternate separator and must be normalized BEFORE traversal checks (Vite's CVE-2025-62522 is the canonical regression). |
| Case sensitivity | Compare canonical paths after normalization. | NTFS is case-insensitive by default; `.Claude/Settings.json` resolves to the same inode as `.claude/settings.json`. Compare with `.resolve()` then `.as_posix().lower()` or `os.path.normcase()`. ([Python pathlib docs](https://docs.python.org/3/library/pathlib.html), [discuss.python.org thread](https://discuss.python.org/t/getting-the-correct-file-path-on-case-insensitive-systems/43817)) |
| 8.3 short names | OWASP's IIS short-name finding shows `PROGRA~1` style aliases enumerate or reach long-named files even when the long form is blocked. ([OWASP Path_Traversal](https://owasp.org/www-community/attacks/Path_Traversal)) | Resolve via `Path.resolve(strict=False)` and re-check containment AFTER resolution; `~` in any path segment is a smell. |
| Alternate Data Streams (ADS) | NTFS allows `file.md::$DATA` or `file.md:hidden.txt` to address a stream attached to the file. ([OWASP ADS](https://owasp.org/www-community/attacks/Windows_alternate_data_stream)) | Reject any path segment containing `:` other than the drive-letter colon at position 1; reject `::$DATA`-style suffixes outright. |
| Reserved device names | `CON`, `PRN`, `AUX`, `NUL`, `COM1-9`, `LPT1-9` (and these names with **any** extension, e.g. `NUL.txt`) are device aliases in legacy code paths. ([Microsoft Win32 docs](https://learn.microsoft.com/en-us/windows/win32/fileio/naming-a-file)) | Reject these basenames (case-insensitive) regardless of extension; on writes especially. |
| Symlinks / junctions / mount points | OWASP recommends "refuse symlinks" or resolve and re-validate containment. | Windows has both NTFS symlinks AND junction points / reparse points; `os.path.islink` does not always report junctions. Use `Path.is_symlink()` AND check for reparse points via `os.lstat` `st_file_attributes & FILE_ATTRIBUTE_REPARSE_POINT`. |
| Absolute paths in user input | Reject anything that resolves outside the root. | A bare drive letter (e.g. `C:foo` without `\`) is a *relative* path on the current dir of `C:` — surprising but real. Always `Path(...).resolve()` then compare to the resolved root. |

The general OWASP rule: **canonicalize the input first, then assert containment** (e.g., `resolved_path.is_relative_to(root.resolve())`). Decisions made on the un-resolved string are repeatedly bypassed.

### 2c. Upload, content-type, and response best practice (OWASP File Upload Cheat Sheet)

- **Trust neither the `Content-Type` header nor the file extension; both are trivially spoofed.** Validate magic bytes for binary uploads (PNG/JPG/SVG in our allowlist). ([OWASP File Upload](https://cheatsheetseries.owasp.org/cheatsheets/File_Upload_Cheat_Sheet.html))
- **Set `X-Content-Type-Options: nosniff` on every response** that serves user-controlled bytes, plus `Content-Disposition: attachment` for non-rendered types — defends against PDF/Flash content-hijacking and stops browser MIME guessing. ([OWASP File Upload](https://cheatsheetseries.owasp.org/cheatsheets/File_Upload_Cheat_Sheet.html))
- **Enforce body-size limits at TWO layers.** OWASP and the FastAPI community both recommend a reverse-proxy cap (`client_max_body_size`) AND an in-app chunked-read cap, because either alone is bypassable (no proxy in dev; in-app cap doesn't help if the framework buffers the whole body first). ([FastAPI discussion #8167](https://github.com/fastapi/fastapi/discussions/8167), [betterstack guide](https://betterstack.com/community/guides/scaling-python/uploading-files-using-fastapi/))
- **SVG is a code-execution vector.** SVG is XML and can carry `<script>`. If SVG is in the allowlist, serve it with `Content-Disposition: attachment` and `Content-Security-Policy: default-src 'none'`, or sanitize on read. ([OWASP File Upload](https://cheatsheetseries.owasp.org/cheatsheets/File_Upload_Cheat_Sheet.html))
- **For state-changing requests on localhost, validate `Origin` / `Sec-Fetch-Site`.** This is the OWASP-recommended layered defense against both CSRF and DNS rebinding. ([OWASP CSRF cheat sheet](https://cheatsheetseries.owasp.org/cheatsheets/Cross-Site_Request_Forgery_Prevention_Cheat_Sheet.html))

## 3. Implications for the spec (concrete, actionable)

1. **`safe_resolve` contract (Windows-aware).** The function MUST: (a) reject any input containing NUL bytes, `::`, `$DATA`, or basenames matching the reserved-device list (case-insensitive, with or without extension); (b) normalize separators (`/` and `\` both → OS native) BEFORE traversal checks; (c) `Path(root, user_input).resolve(strict=False)`; (d) assert `resolved.is_relative_to(root.resolve())` using **case-folded** comparison on Windows (`os.path.normcase`); (e) `lstat` the resolved path and refuse if `S_ISLNK`, `is_junction`, or `FILE_ATTRIBUTE_REPARSE_POINT` is set. This addresses Vite-style separator bypass, NTFS case-collision, ADS, reserved names, and reparse points in one pass.
2. **Extension allowlist alone is insufficient.** For `.png/.jpg/.svg`, validate magic bytes on `PUT`. For `.svg`, additionally serve with `Content-Disposition: attachment` and a strict CSP, OR sanitize SVG on write (strip `<script>`, event handlers, external refs).
3. **Always-on response headers.** Every `GET /api/file` and `GET /api/tree` response: `X-Content-Type-Options: nosniff`, `Cache-Control: no-store` for editable files, `Content-Security-Policy` for HTML responses (the React app shell only).
4. **Two-tier body-size cap.** Spec a hard cap (e.g., 5 MiB for text artifacts, 20 MiB for images) enforced both via a starlette/FastAPI middleware that streams + counts AND a documented note for any future reverse proxy. Reject with `413` before reading the full body.
5. **Origin / Host validation on every state-changing route** (`PUT /api/file`, `POST /api/promote`, `DELETE /api/promote`, `POST /api/regen-prompt`). Accept only `Origin: http://127.0.0.1:8765` (or the configured dev origin) and `Host: 127.0.0.1:8765`. This is the cheap, OWASP-aligned defense against DNS rebinding and accidental cross-origin POSTs from a malicious tab.
6. **Tree exposure is a positive list, not a negative list.** Vite's repeated regressions (deny-list bypassed by `\`, by `.`, by `?import`) argue that EXPOSED_TREE should be expressed as "these directories under repo_root and these only", not "everything except `.git`, `.venv`, `node_modules`". The current spec is positive-list — keep it that way and resist the inevitable feature-request to add a deny-list.
7. **403 vs 404 on out-of-sandbox paths.** OWASP and existing CVE post-mortems lean toward `404` for both "not found" and "outside sandbox" so an attacker cannot enumerate the layout via 403/404 differentiation. The current spec text mentions "403/404" — recommend pinning to a single status (404) to remove the side-channel.
8. **Audit-log every write, including rejections.** Append an entry to `.audit/adhoc_agents/{date}/{task_id}/events.jsonl` on each `PUT` outcome (`fs.write.accepted` / `fs.write.rejected` with reason). This gives the user a tamper-evident log if a malicious page ever reaches `127.0.0.1:8765`.

## 4. Open questions surfaced

- **Authentication on localhost?** Current spec is unauthenticated. Should the webapp require a per-launch token (printed to the CLI on startup, sent via `Authorization` header or query param on first load) to defeat any cross-origin tab that bypasses Origin checks? OWASP's DNS-rebinding mitigation list ranks "require auth" above "validate Host/Origin" for sensitive endpoints.
- **Are symlinks INSIDE the EXPOSED_TREE allowed?** The user said "symlinks refused" — does that mean refuse symlinks anywhere on the resolved path (the strict reading), or only refuse symlinks pointing OUTSIDE the tree? The strict reading is safer; confirm.
- **Should `.git/`, `.venv/`, `node_modules/` be silently invisible** even when nominally inside the repo root, or returned as 404s? The current spec lists positive paths only, so this should be enforced by the positive list, not by additional deny rules — but worth re-stating.
- **Junction-point enumeration on Windows.** Some user repos have `node_modules` containing junctions to a global package store (pnpm). If a researcher / spec author opens such a repo, junctions inside the tree could legitimately point outside. Decision: refuse, or silently skip with audit log?
- **SVG policy.** Allow with sanitize-on-write, allow with response hardening, or drop SVG from the allowlist entirely? Lowest-risk is "drop"; highest-utility is "sanitize on write."
- **Per-file size caps vs global caps.** A single 20 MiB image is fine; a thousand 5 MiB markdown files in a single PUT-loop is not. Should there be a per-IP rate cap on `PUT`?
