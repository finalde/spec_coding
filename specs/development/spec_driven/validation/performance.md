## Scope

Performance checks derived from NFR-1, NFR-2, NFR-3 and the scale ceilings in NFR-15 (â‰¤50 projects, â‰¤200 files per project, individual files <500 KB, hard ceiling <2 MB). All checks target a typical developer laptop running localhost; no remote/CI performance infra is assumed.

## Conventions

- "p95 < N ms" means: collect at least 100 samples (or as specified), compute the 95th percentile of the wall-clock latency, and the value must be strictly below N ms.
- "Hard pass/fail budget" means a regression here blocks release.
- "Observe-only metric" means the value is recorded but does not gate release; large changes still warrant investigation.
- "Cache cleared" means: backend has no in-memory cache (per FR-3); for frontend, clear the browser cache and reload with DevTools "Disable cache" enabled.
- All measurements are taken with the backend bound to `127.0.0.1:8765` (NFR-7) and no other heavy processes contending for CPU.
- The backend computes the tree on every request (FR-3) so warm/cold differ only in OS-level filesystem cache state. Each measurement is preceded by a fixture-touch step (`find <fixture> -type f -exec cat {} >/dev/null \;` or PowerShell equivalent) to warm the OS cache when "warm" is required, or by a `sync; echo 3 > /proc/sys/vm/drop_caches` (Linux) / process restart on Windows when "cold" is required.

## Hard pass/fail budgets

### PERF-01 â€” `GET /api/tree` at locked scale

- **Budget:** p95 < 200 ms (derives directly from NFR-1).
- **Workload:** 50 projects spread across 2 task_types (`development` and `ai_video`, 25 projects each). Each project has 5 stage subfolders, each populated with 40 files (200 files per project, 10000 files total across projects). File contents are 4 KB of synthetic markdown each (small enough not to stress disk read but large enough to be realistic). Plus Section 1: `CLAUDE.md` (4 KB), 8 agent files in `.claude/agents/`, 6 skill folders each with a `SKILL.md`.
- **Setup:** generator script `tests/fixtures/perf_tree/build.py` (pseudocode below) writes the fixture under `tests/fixtures/perf_tree/repo/` containing the synthetic `CLAUDE.md`, `.claude/`, and `specs/` trees. The harness launches the backend with `REPO_ROOT` overridden (or via a CLI flag) to point at the fixture root.
- **Method:** warm the OS cache once. Issue 100 sequential `curl http://127.0.0.1:8765/api/tree -o NUL` requests. Record per-request wall-clock from `curl -w '%{time_total}'`. Discard the first 5 as warm-up. Compute p50/p95/p99 over the remaining 95.
- **Pass criterion:** p95 strictly < 200 ms. Also report p50 and p99 for context.
- **Failure escalation:** an out-of-budget result implies the per-request filesystem walk is too expensive at the locked scale and either (a) the implementation must add caching despite FR-3 saying it shouldn't be needed, (b) the walk has accidental quadratic behavior, or (c) NFR-1 itself needs renegotiation. Halt and surface to the spec author before adding caching.
- **Regression-check approach:** run manually before every release, and additionally any time the tree-building code in `libs/tree.py` (or equivalent) changes.

### PERF-05 â€” Initial app load (cold-cache) on localhost

- **Budget:** end-to-end wall clock < 2000 ms (derives directly from NFR-3).
- **Workload:** the released app served by `make run` against the actual `spec_driven` repo (same scale as PERF-02). Browser: latest stable Chromium (per NFR-14). Browser cache disabled / cleared before each run.
- **Setup:** build the React bundle (`make run` performs the build then starts FastAPI). Open Chromium with a clean profile (`--user-data-dir=/tmp/spec_driven_perf_profile` or PowerShell equivalent), DevTools open with "Disable cache" checked under Network.
- **Method:** any one of the following three is acceptable; pick one and stick to it for repeatability. Take the median of 5 runs (not p95 â€” small sample, wall-clock-dominated).
  - **A. Chrome DevTools Performance trace.** Record from `navigationStart` to `loadEventEnd` plus the first `/api/tree` and first `/api/file` responses. Read the value from the trace summary.
  - **B. Lighthouse local run.** `npx lighthouse http://localhost:8765/ --only-categories=performance --preset=desktop --output=json`. Use the `interactive` metric (Time to Interactive) as the wall-clock proxy.
  - **C. Puppeteer script.** Navigate to `http://localhost:8765/` with `waitUntil: 'networkidle0'`; measure from `Date.now()` before navigation to the resolution of `page.goto`. Sufficient because `networkidle0` waits for no in-flight network for 500 ms, which captures the auto-redirect to `/projects/.../spec.md`, the `/api/tree`, and the first `/api/file`.
- **Pass criterion:** median of 5 cold-cache runs strictly < 2000 ms. Hard fail if any single run exceeds 3000 ms (catches outliers).
- **Failure escalation:** out-of-budget implies the bundle is too large, the auto-redirect chain is too slow, or rendering Shiki + the spec file blocks too long. Inspect the JS bundle size (target <500 KB gzipped for the main chunk), check whether Shiki is being loaded eagerly when it could be lazy. Halt before release; surface to the spec author if remediation requires architecture change.
- **Regression-check approach:** run manually before release. Bundle size can additionally be tracked by a one-line `ls -la projects/spec_driven/backend/static/assets/*.js` after each build.

### PERF-06 â€” Concurrent tree-walks

- **Budget:** p95 < 500 ms across 10 concurrent requests; zero non-2xx responses (implicit from the no-cache design in FR-3 â€” the spec does not require concurrency, but a regression where parallel walks crash or serialize badly should be caught).
- **Workload:** the same locked-scale fixture used by PERF-01 (`tests/fixtures/perf_tree/repo/`).
- **Setup:** reuse PERF-01 fixture and harness.
- **Method:** issue 10 concurrent `GET /api/tree` requests using `xargs -P 10` over `curl` (or PowerShell `ForEach-Object -Parallel`). Repeat the burst 10 times back-to-back, each burst preceded by a 200 ms idle. Record per-request latency and HTTP status. Compute p95 over the 100 samples; check for any non-200 responses.
- **Pass criterion:** p95 strictly < 500 ms AND zero non-2xx responses across all 100 requests.
- **Failure escalation:** non-2xx responses or p95 blowout under 10Ã— concurrency suggests a global lock (e.g., on a shared `Path.resolve`), file-handle exhaustion, or an event-loop block. Investigate; depending on severity, may or may not block release (the spec does not require concurrency support, but crashes are not acceptable).
- **Regression-check approach:** run manually before release.

- **Method:** open Chrome DevTools â†’ Performance tab â†’ Record â†’ navigate to the file â†’ wait until rendered â†’ stop recording. Inspect the recording for "Long Tasks" (>50 ms warnings) and the longest individual main-thread task.
- **Pass criterion:** no individual main-thread task exceeds 100 ms during the render. Also acceptable: a series of short tasks summing to >100 ms total is fine â€” the user-perceptible criterion is freeze duration, not total work.
