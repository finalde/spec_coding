# Angle: spec-driven parallel audit

Researcher 01. Run `ai_video_management-20260505-002710`. Local-only audit of `projects/spec_driven/` source vs. the ai_video_management revised prompt + interview Q/A.

## 1. What this angle covers

A file-by-file inventory of `projects/spec_driven/` (backend libs, frontend components, markdown views, lib helpers, styles, representative tests, build glue) and a port-action classification for each file:

- **PORT_VERBATIM** ŌĆö copy with only path / string substitutions (`8765` ŌåÆ `8766`, `spec_driven` ŌåÆ `ai_video_management`, allowed-top-level set membership change).
- **PORT_WITH_ADAPTATION** ŌĆö same shape, content diverges (e.g. `tree_walker.py` walks different roots, `regen_prompt.py` honours ai_video novel scope, `Reader.tsx` dispatches new view kinds).
- **NEW** ŌĆö no spec_driven counterpart (`ShotPairView`, `ShotlistTableView`, `ImageRefView`, `subTypeBadge`, etc.).

Stage 4 spec writes one FR per PORT_WITH_ADAPTATION row and per NEW row; stage 6 implementation can proceed file-by-file.

Notable up-front discoveries that contradict the revised-prompt's outline (Stage 4 must reconcile, not silently follow the prompt):

- **No `components/views/` subfolder.** spec_driven keeps view modules in two locations: `components/` (`QaView.tsx`, `Editor.tsx`, `Reader.tsx`, `Sidebar.tsx`, etc.) and `markdown/` (`renderer.tsx`, `CodeView.tsx`, `JsonlView.tsx`, `ImagePlaceholder.tsx`). The revised prompt's nested `components/views/` should NOT be invented ŌĆö keep the existing two-folder split for parity.
- **No `render_views.py` / view-mode registry on the backend.** spec_driven dispatches view choice purely on the frontend (extension + filename heuristics in `Reader.tsx`). The revised prompt's proposed `backend/libs/render_views.py` is not needed ŌĆö view selection stays client-side.
- **Backend module names already differ from the revised prompt's outline.** Actual files: `api.py`, `api_security.py`, `exposed_tree.py`, `file_reader.py`, `file_writer.py`, `promotions.py`, `regen_prompt.py`, `repo_root.py`, `safe_resolve.py`, `stages.py`, `tree_walker.py`. The revised prompt's `server.py / tree.py / files.py / security.py / render_views.py` should NOT replace these ŌĆö keep the spec_driven names so security audits and tests port cleanly.
- **Two separate React error boundaries** (`ParseFallback.tsx` for generic parse-on-render components, `QaErrorBoundary.tsx` for the `QaView`). Both ship; do not merge.
- **The frontend has a `react-router-dom` v6 router** with project / stage routes (`/project/:type/:name`, `/stage/:type/:name/:stage`) ŌĆö not just `/file/*`. ai_video routes need an analogous shape.
- **`/api/stages` already exists** and emits `Stages.to_payload()` from `stages.py`. The `RegeneratePanel` falls back to a hard-coded `DEFAULT_STAGES` constant if the call fails ŌĆö that fallback is the same six stages and must mirror in ai_video_management with identical semantics.
- **`autonomousMode.ts` localStorage key** is `spec_driven.autonomous_mode.v1`. Per `CLAUDE.md` ┬¦ Regeneration prompts the key is project-named, so ai_video_management uses `ai_video_management.autonomous_mode.v1` (not a shared key ŌĆö keeps simultaneous-run behaviour clean).
- **`ALLOWED_EXTENSIONS` already contains `.png`, `.jpg`** (read-only); `file_writer.py` lines 75ŌĆō76 explicitly raise `UnsupportedExtension` for image extensions. The "image extensions are not writable" hard constraint #4 in the revised prompt is already in place ŌĆö port verbatim.
- **`CANONICAL_STAGES` in `exposed_tree.py` and `stages.py`** drive a lot of behaviour (sidebar grouping, promotions allowlist, regen-prompt section emission). They must be re-pointed to `ai_video`-relevant stages but the *list of folders* (`interview`, `findings`, `final_specs`, `validation`) is identical because both task types share the spec-pipeline shape. Stage 6 (`ai_videos/{name}/`) is the only divergence and is already carved out (no v1 promotion).

## 5. Cross-cutting interface contracts to preserve

These are interfaces that span backend Ōåö frontend or component Ōåö component. Stage 4 spec must call them out so stage 6 doesn't accidentally drift.

1. **Tree node shape ŌĆö `{type: "section"|"directory"|"file", name: string, path: string, children?: TreeNode[]}`.** Sidebar walks `node.children` uniformly regardless of node `type`. New optional fields (`subType`) are additive and ignored by Sidebar's recursive walker ŌĆö only the new `SubTypeBadge` reads them. Do NOT introduce a parallel "ai_video tree node" type.
2. **`/api/file` payload ŌĆö `{path, content, encoding: "utf-8" | "base64", bytes, mtime, mtime_http}`.** ImageRefView depends on `encoding === "base64"` for image files, base64 string usable directly as `<img src="data:image/png;base64,ŌĆ”">`. No backend changes needed.
3. **Pin marker in promoted.md sidecar ŌĆö `<!-- pin source=<file> id=<id> -->\n<body>\n`.** `Promotions._serialize` writes this; `extractPinnedIds` in `Reader.tsx` reads it; `regen_prompt.py._render_stage_section` re-inlines it at regen time. Stage 4 must not invent a different sentinel.
4. **PIN_MARKER regex `^(FR|NFR|AC|SYS|OQ)-(\d+[a-z]?)\.?$`** in `markdown/renderer.tsx`. ai_video specs already use this convention; do not narrow or widen it.
5. **`PinContext` shape passed to `<Renderer>`** ŌĆö `{projectType, projectName, stageFolder Ōłł {"interview","findings","final_specs","validation"}, pinnedIds, onPin(id), onUnpin(id)}`. Keep verbatim; it is wired through `Reader.tsx` based on the `specs/{type}/{name}/{stage}/...` URL classification, NOT on the file under `ai_videos/*` (which is intentionally unpinnable in v1).
6. **Regen-prompt assembly schema** ŌĆö `{prompt, warning?, selected_stages_count, follow_ups_count, autonomous, bytes}`. UI displays `warning` when soft limit (50 KB) is exceeded; throws on hard limit (1 MB). New `scope` field is additive. The header line `# EXECUTION MODE: AUTONOMOUS|INTERACTIVE` MUST stay byte-identical to the `CLAUDE.md` ┬¦ Regeneration prompts contract.
7. **`/api/promote` body shape** ŌĆö `{project_type, project_name, stage_folder Ōłł ALLOWED_STAGE_FOLDERS, source_file, item_id, item_text}`. `project_type="ai_video"` flows through unchanged. Stage 6 in ai_video has no v1 promotion (per interview lock + ai_video.md rule 10), so the SPA does not surface a pin button on `ai_videos/*` paths.
8. **Origin/Host gate** ŌĆö `(method, path) Ōłł GUARDED_ROUTES = {(PUT, /api/file), (POST, /api/regen-prompt), (POST, /api/promote), (DELETE, /api/promote)}`. Vite dev proxy MUST rewrite `Origin` to the backend's bound URL (`http://127.0.0.1:8766`) AND `changeOrigin: true` rewrites `Host` ŌĆö both are required, per spec_driven follow-up 006. Foreign / missing ŌåÆ 403; loopback aliases (`localhost` Ōåö `127.0.0.1` at the bound port) admit because they resolve to the same socket.
9. **ParseFallback wrapping convention** ŌĆö every parse-on-render component (`MarkdownView` / `JsonlView` / `CodeView` / `QaView` / NEW `ShotPairView` / NEW `ShotlistTableView` / NEW `ImageRefView`) MUST be inside `<ParseFallback rawText={file.content} componentName="...">`. This is the project-output rule `agent_refs/project/development.md` enforces. Stage 6 cannot ship a new view without this wrapper.
10. **`extOf(path)` heuristic** ŌĆö Reader.tsx switches on `extOf(path)` returning `.png|.jpg|.md|.jsonl|.json|.yaml|.yml|.txt`. New view dispatch (ShotPair / Shotlist / ImageRef) is layered ON TOP of `.md` ŌĆö i.e. all three new views are markdown files; the `isMarkdown` check still passes; we just route to ShotPair / Shotlist / ImageRef BEFORE falling through to `<Renderer>`. Order in the JSX dispatch matters: most-specific first.
11. **`SafeResolver` allowlist** ŌĆö `_ALLOWED_TOP_LEVEL` MUST include `ai_videos` (and drop `projects`); `_EXCLUDED_TOP_LEVEL` reused verbatim. Symlink rejection / mixed-slash / Windows-reserved / `~` / `%` / `..` / NUL byte all reused. Any new path probe in `/api/file` flows through `resolver.resolve(path)` ŌĆö the security model is gate-once, never gate-twice.
12. **localStorage key namespacing** ŌĆö `ai_video_management.autonomous_mode.v1`, NOT shared with `spec_driven`. Both apps run simultaneously; sharing the key would couple their autonomous toggles confusingly.
