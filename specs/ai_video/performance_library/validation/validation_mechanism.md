# Validation-mechanism level — 检验视频 + 双模型 + 1–5 评分门的设计校验与完整规范

本级校验的是**库自身的验证回路**：FR-11 控制变量检验视频 + FR-13 双模型 1–5 评分门 + 迭代修正回路。理由（用户前提）：表演是 AI 视频模型服从性最差的维度，**只有经实证验证的条目才算资产**——所以验证回路本身的健全度，就是这个库的核心价值。本文件把验证回路完整规范到「stage 6 能照着跑」的颗粒度，并指出 spec 留下的设计洞。

本文件锚定 OQ-1（精品微表情派，过火评分偏严，已决议）与 OQ-3（pilot-first，stage 6 先做压抑隐忍，已决议）。

## §1 控制变量完整性（FR-11）—— 枚举所有必须钉死的变量

FR-11 的契约是「唯一变量 = 本 entry 的锁定文本块」。要让「唯一」成立，下列变量**全部必须钉成全库常量**。spec FR-11 列了「固定 actor / 场景 / 机位 / 时长」，但留了若干未钉死的变量，逐一标注。

### 1.1 必须钉死的变量清单（实测变量隔离矩阵）

| # | 变量 | spec 是否已钉 | 必须钉成 | 风险（若不钉） |
|---|------|--------------|----------|----------------|
| V1 | **测试演员**（`_actors/actor_NNNN`） | ✅ FR-11 已要求「全库统一少数几个中性试镜演员」 | 见 §1.2 钉成 1 主 + 1 备 | 不同脸型/年龄使表情可识别度不可比 |
| V2 | **场景/背景** | ✅ 「统一中性背景」 | 纯灰/浅灰无纹理墙，无道具、无窗、无家具 | 背景叙事线索会替模型「补」情绪，污染「情绪可识别」 |
| V3 | **景别** | ⚠️ FR-11 只说「统一景别+机位」，未给具体值 | **近景（人物占画约 60–70%，肩部以上）** | 远景吃不到微表情（FACS 底表的眉/眼/唇动作不可辨）；大特写又丢肢体载体 |
| V4 | **机位/角度** | ⚠️ 同上未给值 | **正面平视、水平视线高度、镜头静止（无运镜）** | 仰/俯角与运镜本身携带情绪语义，污染达意判定 |
| V5 | **时长** | ✅ FR-11 举例 5s | **统一 5s**（钉死，非「如」） | 时长变化改变模型能展开的动作弧，长短不可比 |
| V6 | **光照** | ❌ **spec 完全未钉**（设计洞） | **均匀中性白光（约 5000–5600K），无明显方向硬光、无逆光、无冷暖调倾向** | dossier 明示「冷调=压抑/暖调=新生」是情绪载体——光照不钉，光本身就在演情绪，直接污染「情绪可识别」。**这是 §1 最严重的漏钉项。** |
| V7 | **画幅比** | ❌ **spec 未钉**（设计洞） | **9:16**（继承 ai_video.md 默认） | 16:9 vs 9:16 改变构图与人物占画比，破坏 V3 |
| V8 | **模型版本** | ⚠️ NFR-5 提到要记录，但未要求「同批次锁同版本」 | 同一验证批次内 Kling/Seedance **各锁一个版本号**，记入元数据（见 §2、§7） | 跨版本渲染的分数不可比；这是 re-validation 的锚点 |
| V9 | **使用模式**（纯文本 / 起始帧） | ✅ FR-13 已作为记录字段 | 见 §5：mode 是**记录变量**，不是钉死变量——同一 entry 可两种模式各测，但每条渲染记录单一 mode | 若同一渲染混用模式，A/B 数据不可用 |
| V10 | **prompt 结构骨架** | ❌ **spec 未钉**（设计洞） | 检验视频 prompt 的**固定模板**：除「锁定文本块」槽位外，演员描述符 / 场景句 / 机位句 / 时长 / 画幅**逐字节相同**，Kling 版与 Seedance 版之间也仅模型适配措辞不同 | 若每条 entry 现场重写 prompt 骨架，「唯一变量」名存实亡——骨架的措辞差异会盖过表演描述差异 |
| V11 | **种子 / 随机性** | ❌ spec 未钉（已知不可控） | **不可钉**（Kling/Seedance 无暴露固定 seed）；缓解：见 §3.4「单条多次渲染」建议 | 同 prompt 两次渲染结果有方差，单次打分可能误判——这是评分门最大的统计学软肋 |

### 1.2 提议的全库固定测试台规格（canonical test-rig spec）

落库为 `_performances/_testrig.md`（下划线前缀的库级常量文件，与 `_actors`/`_voices` 同级思路），stage 6 开篇先建立、所有 entry 引用它：

## §2 评分元数据 schema（FR-13）—— `## 实测与验证` 的精确结构

FR-13 列了字段但未给可机检的结构。下面定死 schema：**每个「渲染记录」是 (模型 × 模式) 的一次实测**，一条 entry 的 `## 实测与验证` 是一组渲染记录 + 一个派生判定。

### 2.1 schema 定义

```markdown
## 实测与验证

validation_status: draft | pending_review | validated | needs_revision | blocked
validation_round: 0            # 已完成的修正轮数，0=首测，封顶 3
passing_models: []             # 通过门的模型列表，如 [seedance]；空=未通过

### 渲染记录
| 模型 | 版本/日期 | 模式 | 表演达意 | 情绪可识别 | 是否过火 | 通过 | 实测笔记 |
|------|-----------|------|----------|-----------|----------|------|----------|
| kling | <版本>@<YYYY-MM-DD> | 纯文本/起始帧 | 1-5 | 1-5 | 1-5 | ✓/✗ | <哪些词 land/被忽略> |
| seedance | <版本>@<YYYY-MM-DD> | 纯文本/起始帧 | 1-5 | 1-5 | 1-5 | ✓/✗ | ... |
```

字段语义（钉死，避免作者各自解读）：
- **表演达意 1–5**：锁定文本块里的物理动作是否被渲出（1=几乎被忽略 / 5=全部 land）。映射失败模式：under-rendering。
- **情绪可识别 1–5**：盲看（不告知目标情绪）能否认出目标情绪（1=认不出/认错 / 5=一眼即中）。映射失败模式：ambiguity。
- **是否过火 1–5**：1=自然克制（电影感）/ 5=严重过火（土味浮夸、僵硬）。**越低越好。** 锚点按 OQ-1「精品微表情派」偏严判。映射失败模式：高强度天花板的 stiffness。
- **通过（单模型）**：该行 `表演达意≥4 且 情绪可识别≥4 且 过火≤2` ⇒ ✓。
- **passing_models**：所有 ✓ 行的模型集合，由表派生（NFR-4 可 grep）。
- **版本/日期**：V8 锚点；re-validation 的判据（§7）。

### 2.2 具体填例（压抑隐忍·强度3·内敛·眼神，假想 perf_0003）

```markdown
## 实测与验证

validation_status: validated
validation_round: 1
passing_models: [seedance]

### 渲染记录
| 模型 | 版本/日期 | 模式 | 表演达意 | 情绪可识别 | 是否过火 | 通过 | 实测笔记 |
|------|-----------|------|----------|-----------|----------|------|----------|
| kling    | kling-2.1@2026-06-13 | 纯文本 | 4 | 4 | 3 | ✗ | 「下睑收紧」land，但「双唇相压抗住嘴角」被放大成抿嘴撇嘴，强度3被演成强度4，过火3卡线 |
| seedance | seedance-2.0@2026-06-13 | 纯文本 | 5 | 4 | 1 | ✓ | 上下脸冲突（眼泄漏/嘴平复）完整呈现，克制自然，眼神先垮 land，符合内敛三层模型 |

判定: ≥1 模型（seedance）三轴达标 ⇒ validated。
消费提示: 该条在 Seedance 上 land；Kling 上过火卡 3，若消费者目标渲染器为 Kling，需配起始帧或降一档强度引用。
```

> 设计要点：**「消费提示」是新增的强制行**（spec 漏了）——见 §3.3，解决「Seedance 过、消费者却用 Kling」的盲区。

## §3 通过规则健全度（FR-13）—— 压力测试与精化建议

规则原文：**至少一个模型在「表演达意≥4 且 情绪可识别≥4 且 过火≤2」⇒ pass。**

### 3.1 OR-across-models 是否正确？—— ✅ 正确，但要加约束

dossier 实测：Kling 与 Seedance **反向失败**（Kling 过投射/过火、Seedance 欠演/可能不达意）。因此：
- 要求**两个模型都过** = 把门设到「两类相反失败模式都不犯」，几乎无条目能过，库会空——错误。
- 「≥1 模型过」承认「这条表演在至少一个主流渲染器上是可用资产」——这正是资产的最低定义。✅ 保留 OR。
- **但 OR 不能塌缩成单一 pass/fail 标签**：必须记 `passing_models`（哪个过）。一条「只 Seedance 过」的 entry 与「两个都过」的 entry 是不同等级的资产，消费者需要知道差别。

### 3.2 过火 ≤2 是否该 per-model？—— ✅ 必须 per-model（dossier 直接背书）

过火是 model-correlated（Kling 系统性偏高）。若把两模型过火合并取均值/取一个数，会出现「Seedance 过火1 + Kling 过火4 → 均值2.5 误判可用」的污染。**过火必须按模型分别记、分别参与该模型自己那行的通过判定**（schema §2.1 已落实：每行独立判 ✓）。spec FR-13 已写「过火按模型分别记」，本级确认其健全并钉进逐行判定。

### 3.3 「Seedance 过、消费者用 Kling」盲区 —— ⚠️ spec 的真实设计洞，必须补

场景：entry 仅 Seedance 行 ✓，passing_models=[seedance]。消费者在某剧目用 Kling 渲染该 shot ⇒ 拿到的是该 entry 在 Kling 上**未通过**的表演（很可能过火）。当前 spec 不强制 entry 暴露 per-model 可用性给消费者，消费者会盲用。

**精化建议（3 条，按强制度排序）：**
1. **【强制】** `## 实测与验证` 增 `passing_models` 字段（§2.1 已含）+ `消费提示` 行（§2.2 已示范），显式写明「在哪个渲染器上 land / 在哪个上需起始帧或降档」。消费者 reference 前必读此行。
2. **【强制】FR-10 引用契约联动**：reference-handle 头建议带可选渲染器提示，如 `表演请参考: _performances/perf_0003 (seedance-validated)`，使「这条只在 Seedance validated」在 shot 顶部即可见。本规则回写 ai_video.md 时一并落（FR-10 已约定回写）。
3. **【建议】** 库 README 写明默认消费假设：未标注 `passing_models` 含目标渲染器时，消费者应优先选 entry 通过的那个模型来渲该 shot；不能时降一档强度引用或补起始帧。

### 3.4 单次渲染的统计学软肋 —— ⚠️ V11（seed 不可控）的缓解

模型有渲染方差（同 prompt 两次不同），单次单帧打分可能误判 borderline 条目。**缓解建议（不强制，成本权衡）：**
- 对**判定卡线**（任一轴恰好 =4 或过火恰好 =2）的渲染，**同 prompt 渲 2 次取较差的一次**作判定（保守，宁可漏过不可错收）。非卡线条目单次即可。
- 此规则只对 borderline 触发，避免全库 2× 渲染成本。

### 3.5 ≥4 / ≤2 阈值本身 —— ✅ 与可控带证据一致

dossier：可控带在强度中段，两端（micro 丢、extreme 僵）失败。达意/可识别 ≥4（满分5）要求「清晰 land」而非「勉强可见」，过火 ≤2 排除中高过火——阈值与「精品微表情派」（OQ-1）锚点一致，保留。**唯一补充**：对 `intensity=1`（痕迹/微表情）条目，dossier 警告「micro 常被丢」，达意 ≥4 可能系统性难达标；建议 stage-6 实测后若强度1普遍卡达意，**允许强度1的 entry 把 mode 默认设为起始帧**（用峰值静帧兜底微表情），而非放宽阈值。放宽阈值会破坏资产定义；改 mode 不破坏。

## §4 迭代回路（FR-13 + CLAUDE.md 迭代上限）

### 4.1 回路定义

```
撰写(physical-action body) → 生成检验视频 prompt(控制变量, §1.2 模板)
  → 用户双模型渲染 + 三轴打分(§2 schema)
  → 派生 passing_models
     ├─ 非空 ⇒ validation_status = validated，写消费提示，结束
     └─ 空   ⇒ validation_status = needs_revision，进入修正：
            诊断失败轴(达意↓ / 可识别↓ / 过火↑) → 按 §4.2 改写 body
            → validation_round += 1 → 回到「生成检验视频 prompt」重测
```

### 4.2 失败轴 → 修正动作映射（让「re-author」不靠玄学）

| 失败轴 | 诊断 | 修正动作 |
|--------|------|----------|
| 表演达意 <4 | 物理动作没被渲出 | 词汇更具体化（裸描述→FACS 底表的命名肌肉动作 + 主动动词）；拆 beat（避免多动作同帧堆叠，dossier 顺序staging 证据）；加视线锚点 |
| 情绪可识别 <4 | 渲出了但认不出/认错 | 补判别器 AU（如真笑补 AU6、恐惧/惊讶用 AU4 区分）；内敛条补「上下脸冲突」第三层（FR-7）；强度可能不足，按 FR-8 阶梯升一档 AU 集 |
| 过火 >2 | 高强度天花板 stiffness（多见 Kling） | 降强度副词（「微微/缓缓」）；高强度条改 mode=起始帧（峰值静帧 + 小动作插值，避开文本合成峰值的僵硬）；按 OQ-1 精品派收敛通道数 |

### 4.3 上限与熔断（CLAUDE.md 迭代上限对齐）

- **单条修正封顶 3 轮**（`validation_round ≤ 3`）。第 3 轮仍 passing_models 为空 ⇒ 该条 `validation_status: blocked`，停止该条迭代，**不阻塞其余 entry**（per-entry 隔离，非整库 halt）。
- **熔断（circuit-breaker）**：若**同一失败轴在连续两轮以同样原因复现**（如两轮都「过火>2 且笔记同因」），立即停该条、标 `blocked`，**不等跑满 3 轮**（CLAUDE.md「同一 issue 跨两轮复现即 halt」）。
- **blocked 后**：写 `## 实测与验证` 的笔记记录卡点与最后一轮分数；emit 事件（§4.4）；升级给用户（autonomous 模式下内联标注判断、继续其余条目，不调 AskUserQuestion）。
- **wall-clock**：单条验证迭代超 30 分钟亦熔断（CLAUDE.md）。

### 4.4 事件流（stage 6 runtime，对齐 CLAUDE.md event schema）

每条 entry 的验证按 CLAUDE.md stage-6 事件契约 emit（work_unit = 一条 entry）：
- 开测：`validation.started`（`work_unit_id=perf_NNNN`, `levels:[validation_mechanism,...]`, `pre_reading_consulted:[...]`）。
- 每个失败轴：`validation.issue.raised`（`issue_id`, `level=validation_mechanism`, `severity`, `description`=失败轴+原因）。
- 通过：`validation.pass`（`work_unit_id=perf_NNNN`）。
- 强度1 等需人工峰值帧确认的：`validation.requires_manual_walkthrough`。
- 修正落盘：`exec.revision.applied`。
- 3 轮/熔断封顶：`pipeline.halted`（`reason`, `work_unit_id=perf_NNNN`）——注意是该条 halt 语义（blocked），整库继续。

## §5 文本 vs 起始帧 A/B（dossier open finding）—— 让库成为自己的数据集

dossier 核心 open finding：**公开无「同表情 文本 vs 起始帧」对照 A/B，我们的控制变量渲染正是这个缺失实验。** 要让数据日后可用，`mode` 不能只是个随手字段。

### 5.1 mode 记录的最小可用规范

- 每条渲染记录的 `模式` 字段取值 **`纯文本` | `起始帧`**（已在 §2 schema）。
- **同一 entry 若两种模式都测**，则 `## 实测与验证` 的渲染记录表含 4 行（kling×2模式 + seedance×2模式），而非 2 行。`passing_models` 仍按行派生。
- **A/B 可用性最小条件（三者缺一不可）：**
  1. 两模式渲染**共用同一锁定文本块、同一测试台（§1.2）**——唯一差异是有无起始帧。起始帧的 Seedream 生成 prompt 必须落盘在 entry 的 `## 起始帧表情` 块（FR-9 ⑤），保证起始帧可复现。
  2. 两模式的三轴分数**同次、同评分者、同盲看协议**打出（避免评分漂移污染对照）。
  3. 记录 `intensity`（A/B 的预期效应随强度变化——dossier 预测高强度起始帧增益最大）。
- **库级聚合**：库 README 或 `_performances/_ab_log.md` 维护一张派生表（可由各 entry grep 出）：`(intensity, model, mode) → 平均三轴 + 通过率`。这张表稳定后即可回答「起始帧到底在哪个强度段、哪个模型上系统性优于纯文本」——库的最高阶研究产出。
- **最小启动**：pilot（压抑隐忍）阶段，至少把**强度1 与强度5 各一条**做 双模式 ×双模型 = 4 行渲染，作为 A/B 种子点（两端是 dossier 预测效应最强处）。

## §6 中文 prompt 校准（OQ-5）—— stage 6 开篇的去风险渲染

dossier/angle 反复警示：**所有公开实测是英文 prompt；中文肌肉描述（「嘴角微微下垂，眉头轻蹙」）是否同样 land 未经验证。** 库全部内容中文（FR-3），这是 v1 的根基性未验证假设。

**强烈建议（low-cost 早期去风险）：stage 6 在批量撰写任何 entry 之前，先跑一条「中文 vs 英文」校准渲染。**

- **校准对象**：取压抑隐忍·强度3·内敛 一条（中段最可控，校准信噪比最高），写**语义等价的中文版与英文版锁定文本块**（同一组 FACS 底表动作，仅语言不同）。
- **控制台**：完全沿用 §1.2 固定测试台，唯一变量 = 语言。双模型各渲中/英两版 = 4 次渲染。
- **判读**：按 §2 三轴打分，比较中/英分差。
  - 若中文分 ≈ 英文分（差 ≤1）⇒ 中文假设成立，照计划批量撰写。
  - 若中文系统性低于英文（达意/可识别差 ≥2）⇒ **触发 spec 级别警报**：需在 entry body 增「中文措辞规约」（如混排 AU 号？改更口语的画面词？），或回 stage-4 调整——这是会改 body 撰写规则的重大发现，autonomous 下内联标注并升级。
- **落盘**：校准结果写 `_performances/_calibration.md`（库级常量文件），README 引用其结论。**这条校准渲染本身是 stage 6 的第一个 work_unit，先于任何 perf_NNNN。**

## §7 复验节奏（NFR-5）—— 版本漂移使结论失效

NFR-5：Kling/Seedance 版本漂移使服从性结论失效；entry 须记验证所用模型版本/日期。本级把「何时触发复验、entry 如何记录被验版本」规范死。

### 7.1 每条 entry 记录的被验版本锚点

- §2 schema 的 `版本/日期` 列即锚点：每行渲染记录绑定一个 `模型版本@YYYY-MM-DD`。
- `validation_status: validated` 只对**记录在案的那个版本组合**成立——validated 不是永久属性，是「截至 <版本/日期> 的 validated」。

### 7.2 复验触发条件（明确枚举，避免「凭感觉复验」）

触发任一即对受影响 entry 置 `validation_status: needs_revision`（实为 needs_revalidation，复用同状态）并重跑 §4 回路：

1. **大版本跳变**：Kling 或 Seedance 主版本号变更（如 kling-2.x→3.0、seedance-2.0→3.0）⇒ **该模型为 passing_model 的所有高强度（intensity≥4）entry 强制复验**（dossier：天花板表现最受版本影响）；中低强度 entry 标 `stale` 提示、按需复验。
2. **过火行为漂移**：实战中发现某模型新版过火基线整体抬升/下降 ⇒ 该模型全部 entry 的过火轴复验。
3. **库扩批**：每次新增情绪批次（OQ-3 的后续 9 情绪）时，抽查既有 validated entry 在当前版本是否仍通过（防静默漂移）。

### 7.3 复验的记录方式（保留历史，不覆盖）

- 复验**追加新渲染记录行**（带新版本/日期），**不删旧行**——旧行是版本对照的历史数据（与 §5 A/B 数据集同源价值）。
- `passing_models` 与 `validation_status` 按**最新版本行**重新派生。
- 若新版本下从 validated 跌为不通过 ⇒ 进 §4 修正回路（封顶 3 轮，同熔断）。
- 库 README 写明本节奏 + 「validated 是版本绑定的，非永久」的免责声明（NFR-5 已要求 README 写复验说明，本节给出可执行细则）。

## §8 本级判据小结（stage-6 validator 可逐条机检）

1. **测试台常量存在且被引用**：`_performances/_testrig.md` 存在，含 §1.2 全 11 类变量的钉死值（含 spec 漏钉的 V6 光照 / V7 画幅 / V10 prompt 骨架）。
2. **prompt 骨架字节一致**：每条 entry 检验视频 prompt 去 `{{锁定文本块}}` 槽位后与测试台模板 byte-diff 为零。
3. **`## 实测与验证` 结构合规**：含 `validation_status` / `validation_round` / `passing_models` / 渲染记录表（每行 6 字段 + 通过 + 笔记）/ 消费提示行；过火 per-model 逐行判定。
4. **通过判定可派生**：passing_models 由渲染表 ✓ 行 grep 派生；validated ⟺ passing_models 非空。
5. **迭代封顶 + 熔断生效**：validation_round ≤3；同轴同因连续两轮即 blocked；blocked 条不阻塞其余条；对应事件已 emit。
6. **A/B 最小可用**：pilot 至少强度1与强度5 各一条做双模式×双模型；mode 记录满足 §5.1 三条件。
7. **中文校准前置**：`_performances/_calibration.md` 在任何 perf_NNNN 之前产出，结论入 README。
8. **复验锚点齐全**：每行渲染记录带 `模型版本@日期`；README 含复验触发条件与「validated 版本绑定」声明。
